- `usage` 页面增强

对应设计：

- [AI与语音消耗记录设计方案](/Users/shaominhe/成交CRM-CodeX版/docs/AI与语音消耗记录设计方案.md)

## 一、P1 目标

本轮 P1 的目标不是继续扩功能，而是让后台运营能稳定看清三件事：

1. 当前哪些账户快没额度了
2. 成本主要集中在哪些供应商、模型、业务场景
3. 模型倍率与路由策略能否被后台稳定维护

本轮验收范围只覆盖以下五项：

- usage 流水筛选增强
- 供应商 / 模型维度展示
- 模型倍率可视维护
- 低余额预警
- 异常高消耗排查

---

## 二、执行前准备

### 2.1 代码与页面

- 已拉到当前仓库最新代码
- 已刷新后台页面缓存
- 本地 bridge 已正常启动
- 浏览器能打开后台地址

示例地址：

```text
http://127.0.0.1:8731/?provider=cloud&bridgeBase=http://127.0.0.1:8788&operatorKey=你的operatorKey
```

### 2.2 云函数

这一轮至少确认下面两个云函数已经重新部署：

- `adminGetAiModelConfig`
- `adminUpdateAiModelConfig`

### 2.3 数据准备建议

为了把 P1 看全，建议至少准备这些测试数据：

- 至少 2 个账户
- 至少 1 个账户接近语音低余额
- 至少 1 个账户接近 AI 低余额
- `usageLedger` 中至少有 1 条 `voice_seconds` 消耗记录
- `usageLedger` 中至少有 1 条 `ai_tokens` 消耗记录
- `usageLedger.meta` 中至少有 `providerKey / providerLabel / model / multiplier / billingMethod`

## 三、先跑自动检查

在仓库根目录执行：

```bash
node admin-web/scripts/check-usage-p1.js
```

通过标准：

- 输出中没有 `FAIL`
- 最后 `failed: 0`

说明：

- 这是静态结构回归脚本
- 它负责检查页面结构、脚本入口、筛选逻辑挂点、云函数倍率字段
- 它不替代浏览器联调，也不替代真云数据验证

## 四、浏览器人工验收

## 4.1 usage 页面基础加载

- [ ] 打开后台后能进入“额度与订阅”页面
- [ ] 页面顶部能看到 `搜索 / 额度类型 / 时间 / 场景 / provider / model` 六类筛选
- [ ] 页面中部能看到 `低余额预警`
- [ ] 页面中部能看到 `来源场景统计`
- [ ] 页面中部能看到 `供应商维度 / 模型维度`
- [ ] 页面中部能看到 `异常高消耗排查`

通过标准：

- 页面不报错
- 控件不缺失
- 无数据时也有空态说明

## 4.2 usage 流水筛选增强

- [ ] 关键词搜索能缩小账户列表
- [ ] `额度类型` 切到 `语音额度` 后，只保留语音口径
- [ ] `额度类型` 切到 `AI 额度` 后，只保留 AI 口径
- [ ] `时间` 切到 `今天 / 近7天 / 近30天` 时，来源统计和右侧流水同时变化
- [ ] `来源场景` 下拉切换时，来源统计卡片与右侧流水同步变化
- [ ] 输入 `providerKey` 关键字后，列表与统计都按 provider 收窄
- [ ] 输入模型名关键字后，列表与统计都按 model 收窄
- [ ] 点击 `重置筛选` 后，`type / time / source / provider / model / 预警` 全部恢复默认

重点核对：

- [ ] 当 provider/model 没有命中任何记录时，筛选条件不会被系统自动清空
- [ ] 当当前选中账户被筛选掉时，右侧详情会自动切到当前结果集中的首个账户，或明确显示未选中

## 4.3 低余额预警

- [ ] 页面能显示 `语音低余额 / AI 低余额 / 双低余额` 三张卡
- [ ] 点击任一预警卡后，会高亮当前卡片
- [ ] 点击后，账户视图数量和列表结果会按该预警口径收窄
- [ ] 再次点击同一卡片，可取消该预警筛选

默认阈值口径：

- 语音低余额：`<= 120 秒`
- AI 低余额：`<= 10000 token`

## 4.4 供应商与模型维度展示

- [ ] 供应商卡片能展示供应商名、AI 消耗、消耗条数、语音消耗
- [ ] 点击供应商卡片后，provider 输入框会自动回填
- [ ] 模型卡片能展示“供应商标签 + 模型名”
- [ ] 点击模型卡片后，会同时锁定 `provider + model`

重点核对：

- [ ] 如果不同供应商下有同名模型，不会被混算到同一张模型卡
- [ ] 切换模型卡后，右侧详情流水也会同步按 provider + model 收窄

## 4.5 异常高消耗排查

- [ ] 页面能显示 `高消耗账户 TOP`
- [ ] 页面能显示 `高消耗场景 TOP`
- [ ] 切换 `type / time / source / provider / model` 后，TOP 列表会同步变化
- [ ] 当当前口径下无明显数据时，能显示空态文案，而不是报错或空白

## 4.6 右侧额度详情一致性

- [ ] 点击左侧不同账户，右侧会刷新到账户自己的额度详情
- [ ] 右侧“最近额度流水”统计口径与左侧当前筛选一致
- [ ] 右侧流水卡中能看到项目、模型、计费方式、原始用量、倍率等信息
- [ ] 技术追踪折叠区能正常展开

重点核对：

- [ ] 右侧明细不会出现“左侧已按 provider/model 过滤，但右侧仍显示全部流水”的错位

## 五、AI 模型配置中心验收

## 5.1 供应商页签

- [ ] 可以打开 `AI 模型配置中心`
- [ ] 默认先落在 `供应商管理` 页签
- [ ] 每个供应商卡片都能看到 `defaultModel`
- [ ] 每个供应商卡片都能看到 `API Key`
- [ ] 每个供应商卡片都能看到 `baseURL`
- [ ] 每个供应商卡片都能看到 `模型倍率（modelPricing）`

## 5.2 模型倍率维护

- [ ] 在任一供应商卡中填写多行 `model=multiplier`
- [ ] 点击 `保存 AI 配置` 后保存成功
- [ ] 刷新页面后，倍率配置仍能被重新读出
- [ ] 非法倍率行不会导致页面崩溃

建议测试样例：

```text
deepseek-v4-flash=1
gpt-5.4=1.8
glm-4.5=1.2
```

非法样例：

```text
deepseek-v4-flash=0
gpt-5.4=-1
broken-line
```

通过标准：

- 合法项能保存
- 非法项不会写成脏配置

## 5.3 路由页签

- [ ] 可以切到 `路由策略` 页签
- [ ] 所有小程序 AI 场景路由都在这里可见
- [ ] 路由可以绑定 `providerKey`
- [ ] 路由可以单独维护 `model`
- [ ] 路由可以配置 `fallbackProviderKey / fallbackModel`
- [ ] 点击 `测试当前配置` 后能拿到明确结果

---

## 六、最小回归顺序

每次再动 `usage` 或 `AI 模型配置中心` 后，优先回归这 6 条：

1. 跑 `node admin-web/scripts/check-usage-p1.js`
2. 打开 `额度与订阅` 页面，看 usage 模块是否完整渲染
3. 试一次 `provider + model` 联合筛选，确认右侧流水同步收窄
4. 点一次 `语音低余额` 或 `AI 低余额` 卡，确认筛选能开关
5. 在 `AI 模型配置中心` 改一条 `modelPricing`，保存后刷新确认回显
6. 在 `路由策略` 页签点一次 `测试当前配置`

## 七、当前这轮的完成定义

这一轮 P1 可以判定为完成，必须同时满足：

- 自动脚本检查通过
- usage 页面五项能力都能在浏览器中看见并可操作
- 模型倍率可保存、可刷新回显
- 路由测试能给出明确通过或失败原因
- 没有出现筛选错位、右侧流水口径不一致、模型同名混算这三类问题

如果自动检查通过，但浏览器联调不通过，优先排查：

1. 页面缓存未刷新
2. `adminGetAiModelConfig / adminUpdateAiModelConfig` 未重新部署
3. `usageLedger.meta` 真实数据里缺 `providerKey / model / multiplier`
4. bridge 仍在跑旧代码
