# 闪录公共组件字段协议

本文档用于承接 [闪录公共组件改造方案](/Users/shaominhe/成交CRM-CodeX版/docs/闪录公共组件改造方案.md)，把后续开发中最容易反复争议的部分先冻结下来：

- 公共组件 `props`
- 组件内部状态 `data`
- 对外事件 `events`
- 组件方法 `methods`
- 页面侧接入协议
- 控制层输入输出协议

目标不是一次把所有实现细节写死，而是先把“边界”和“命名”定下来，避免后续代码成型后再返工。

## 1. 组件命名与文件位置

建议公共组件命名为：

- `components/quick-entry-sheet`

建议共享控制层命名为：

- `services/quick-entry-controller.js`

必要时可继续拆分，但第一阶段先以这两个主体为准。

## 2. 组件职责边界

`quick-entry-sheet` 组件负责：

- 半窗显隐
- UI 渲染
- 用户输入采集
- 派发标准事件
- 根据外部传入状态刷新展示

`quick-entry-sheet` 不负责：

- 直接调用云函数
- 决定是否允许使用闪录
- 决定保存草稿到哪里
- 决定成功后跳项目详情还是留在当前页

这些职责统一归：

- 页面接入层
- `quick-entry-controller`

## 3. 组件 Props 协议

下面的 `props` 建议作为第一版冻结字段。

### 3.1 基础显隐类

```js
{
  visible: Boolean,
  loading: Boolean,
  submitting: Boolean
}
```

字段说明：

- `visible`
  - 必填：是
  - 类型：`boolean`
  - 含义：是否显示闪录半窗
  - 默认值：`false`
  - 写入方：页面

- `loading`
  - 必填：否
  - 类型：`boolean`
  - 含义：是否处于初始化或恢复草稿中的加载态
  - 默认值：`false`
  - 写入方：页面 / 控制层结果映射

- `submitting`
  - 必填：否
  - 类型：`boolean`
  - 含义：是否处于保存提交中
  - 默认值：`false`
  - 写入方：页面 / 控制层结果映射

### 3.2 场景上下文类

```js
{
  context: {
    sourcePage: 'home',
    sourceScene: 'global_entry',
    projectId: '',
    projectName: '',
    lockProject: false,
    readonlyProject: false
  }
}
```

字段说明：

- `sourcePage`
  - 可选值：`home | projects | project_detail | shared_out | mine`
  - 含义：从哪个页面打开

- `sourceScene`
  - 可选值：`global_entry | current_project | shared_record`
  - 含义：打开场景

- `projectId`
  - 含义：当前上下文中的项目 ID
  - 项目页 / 项目详情页打开时可传

- `projectName`
  - 含义：当前上下文中的项目名称
  - 用于组件顶部提示、恢复草稿时说明来源

- `lockProject`
  - 含义：是否默认锁定当前项目，不允许自由切换
  - 第一版建议：
    - 首页：`false`
    - 项目页：`true`
    - 项目详情页：`true`
    - 外发页：`false`
    - 我的页：`false`

- `readonlyProject`
  - 含义：当前上下文项目是否只读
  - 若为 `true`，组件应提示用户不要直接写入该项目，必要时引导重新选择项目

默认值建议：

```js
{
  sourcePage: 'home',
  sourceScene: 'global_entry',
  projectId: '',
  projectName: '',
  lockProject: false,
  readonlyProject: false
}
```

### 3.3 权限与可用性类

```js
{
  capability: {
    canUseQuickEntry: true,
    canUseSpeechToText: true,
    canUseAi: true,
    bindRequiredForWrite: false,
    reasonCode: '',
    reasonText: ''
  }
}
```

字段说明：

- `canUseQuickEntry`
  - 含义：整体是否允许打开和提交闪录

- `canUseSpeechToText`
  - 含义：是否允许语音录入

- `canUseAi`
  - 含义：是否允许 AI 理解、AI 项目匹配、AI 下一步建议

- `bindRequiredForWrite`
  - 含义：当前是否必须先绑定手机号才能正式保存

- `reasonCode`
  - 建议枚举：
    - `bind_required`
    - `speech_exhausted`
    - `ai_exhausted`
    - `write_disabled`
    - `quick_entry_disabled`

- `reasonText`
  - 含义：展示给用户的限制提示文案

说明：

- 权限判断不在组件内部做，组件只消费已经判断好的可用性结果
- 页面接入层应在打开前先判断一次，组件内部再兜底展示一次

### 3.4 业务状态类

```js
{
  mode: 'follow_up',
  draft: {},
  projectOptions: [],
  aiResult: {},
  voiceState: {},
  uiState: {}
}
```

字段说明：

- `mode`
  - 可选值：`follow_up | task | project`
  - 含义：当前闪录模式
  - 默认值：`follow_up`

- `draft`
  - 含义：当前可编辑草稿对象
  - 由控制层统一生成

- `projectOptions`
  - 含义：当前可供选择的项目候选列表
  - 已包含推荐顺序、AI 候选标记、当前选中态所需字段

- `aiResult`
  - 含义：当前 AI 摘要 / 阶段建议 / 下一步建议 / 风险提示

- `voiceState`
  - 含义：当前录音与识别状态

- `uiState`
  - 含义：一些纯展示态的开关，例如是否展开完整理解、是否显示项目搜索

## 4. Draft 字段协议

`draft` 建议统一为一个对象，不按页面定制结构。

```js
{
  mode: 'follow_up',
  sourcePage: 'home',
  sourceScene: 'global_entry',
  projectId: '',
  projectName: '',
  followUpContent: '',
  followUpMethod: '其他',
  followUpDate: '',
  followUpClock: '',
  taskTitle: '',
  taskType: 'other',
  taskDueDate: '',
  taskDueTime: '',
  taskContext: '',
  taskDescription: '',
  selectedProjectId: '',
  selectedProjectName: '',
  selectedProjectLocked: false,
  voicePreviewText: '',
  lastRecognizedText: '',
  lastSourcePage: 'home',
  lastProjectId: '',
  updatedAt: ''
}
```

说明：

- 第一版可以允许不同模式只用其中一部分字段
- 不建议为每种模式维护完全不同的对象结构
- 统一结构更有利于恢复草稿与调试

## 5. Project Options 字段协议

组件接收的项目列表建议统一成以下格式：

```js
[
  {
    id: '',
    name: '',
    client: '',
    stage: '',
    contactText: '',
    nextText: '',
    focusText: '',
    latestSummary: '',
    isAiCandidate: false,
    isAiTopCandidate: false,
    aiCandidateRank: 0,
    isSelected: false,
    isReadonly: false,
    matchDebugText: ''
  }
]
```

说明：

- `projectOptions` 不应再让组件自己拼字段
- 组件只负责展示和选择
- `matchDebugText` 可在调试或灰度阶段显示，生产环境可隐藏

## 6. AI Result 字段协议

```js
{
  summary: '',
  highlights: [],
  risks: [],
  recommendedStage: '',
  stageReason: '',
  nextAction: '',
  recommendedTarget: '',
  recommendedDate: '',
  recommendedTime: '',
  taskDrafts: [],
  sourceType: 'model',
  sourceLabel: '',
  providerLabel: '',
  modelName: '',
  canRegenerate: false,
  status: 'idle',
  errorText: ''
}
```

`status` 建议枚举：

- `idle`
- `loading`
- `ready`
- `candidates`
- `error`

说明：

- `summary` 和 `nextAction` 分属两类结果，但第一版可以挂在同一对象里
- 若后续复杂度升高，再拆成 `aiSummaryResult` / `aiNextSuggestionResult`

## 7. Voice State 字段协议

```js
{
  supported: true,
  recording: false,
  recognizing: false,
  paused: false,
  durationMs: 0,
  previewText: '',
  statusText: '',
  errorCode: '',
  errorText: ''
}
```

说明：

- `recording` 和 `recognizing` 必须是两个独立状态
- 不能只靠一个 `loading` 字段混用
- 这是后续排查“关闭时停没停”“识别中有没有中断”的基础

## 8. UI State 字段协议

```js
{
  showProjectSearch: false,
  showFollowUpDetails: false,
  showFullAiResult: false,
  showSuccessPanel: false,
  showDiscardEntry: false,
  editing: false
}
```

说明：

- `uiState` 只放展示开关
- 不放业务判断结果
- 这样组件和控制层职责更清晰

## 9. 组件对外 Events 协议

建议组件只对外抛标准事件，不直接操作页面路由。

### 9.1 `open`

触发时机：

- 组件真正打开完成后

事件载荷：

```js
{
  context: {},
  mode: 'follow_up'
}
```

作用：

- 页面可用于埋点
- 控制层可用于开始初始化

### 9.2 `close`

触发时机：

- 用户关闭半窗
- 保存草稿关闭
- 放弃本次闪录关闭
- 成功后关闭

事件载荷：

```js
{
  reason: 'cancel' | 'persist_draft' | 'discard' | 'mask' | 'success_close',
  mode: 'follow_up',
  hasPendingChanges: false
}
```

### 9.3 `modechange`

触发时机：

- 用户切换模式

事件载荷：

```js
{
  fromMode: 'follow_up',
  toMode: 'task'
}
```

### 9.4 `fieldchange`

触发时机：

- 任一输入字段变化

事件载荷：

```js
{
  field: 'followUpContent',
  value: '客户周五看方案'
}
```

说明：

- 若后续性能压力大，可改为页面侧主动拉取当前草稿
- 第一版先保留细粒度事件，便于迁移现有逻辑

### 9.5 `voiceaction`

触发时机：

- 用户点击开始录音
- 停止录音
- 重新识别

事件载荷：

```js
{
  action: 'start' | 'stop' | 'retry' | 'cancel'
}
```

### 9.6 `projectselect`

触发时机：

- 用户选择项目

事件载荷：

```js
{
  projectId: '',
  source: 'ai_candidate' | 'manual_search' | 'locked_context'
}
```

### 9.7 `projectsearch`

触发时机：

- 用户输入项目搜索词

事件载荷：

```js
{
  keyword: ''
}
```

### 9.8 `submit`

触发时机：

- 用户点击保存 / 提交

事件载荷：

```js
{
  mode: 'follow_up',
  draft: {},
  context: {}
}
```

### 9.9 `successaction`

触发时机：

- 成功面板点击“继续录入”
- 成功面板点击“查看项目”
- 成功面板点击“关闭”

事件载荷：

```js
{
  action: 'continue' | 'open_project' | 'close',
  successState: {}
}
```

## 10. 组件 Methods 协议

为了方便页面主动控制，建议组件暴露少量标准方法。

### 10.1 `open(payload)`

用途：

- 页面主动打开闪录

参数：

```js
{
  context: {},
  mode: 'follow_up',
  draft: {}
}
```

### 10.2 `close(payload)`

用途：

- 页面主动关闭闪录

参数：

```js
{
  reason: 'cancel'
}
```

### 10.3 `reset(payload)`

用途：

- 页面在成功后继续录入时重置状态

参数：

```js
{
  mode: 'follow_up',
  keepProjectSelection: false
}
```

### 10.4 `applyState(payload)`

用途：

- 控制层计算完后，统一回灌最新视图状态

参数：

```js
{
  draft: {},
  projectOptions: [],
  aiResult: {},
  voiceState: {},
  uiState: {}
}
```

## 11. 页面接入协议

页面侧建议统一维护以下最小状态：

```js
{
  quickEntryVisible: false,
  quickEntryLoading: false,
  quickEntrySubmitting: false,
  quickEntryContext: {},
  quickEntryCapability: {},
  quickEntryState: {}
}
```

页面职责：

1. 判断是否允许进入闪录
2. 生成 `context`
3. 调用控制层初始化
4. 把控制层输出传给组件
5. 接住组件事件后再回调控制层
6. 决定成功后的页面动作

页面不应直接：

- 操作录音状态机
- 操作 AI 返回对象结构
- 各自定义一套草稿结构

## 12. 控制层输入输出协议

`quick-entry-controller` 建议至少提供以下方法。

### 12.1 `createInitialState(payload)`

输入：

```js
{
  context: {},
  mode: 'follow_up',
  capability: {},
  cachedDraft: {}
}
```

输出：

```js
{
  draft: {},
  projectOptions: [],
  aiResult: {},
  voiceState: {},
  uiState: {}
}
```

### 12.2 `updateField(payload)`

输入：

```js
{
  state: {},
  field: '',
  value: ''
}
```

输出：

```js
{
  state: {}
}
```

### 12.3 `handleVoiceAction(payload)`

输入：

```js
{
  state: {},
  action: 'start' | 'stop' | 'retry' | 'cancel'
}
```

输出：

```js
{
  state: {},
  effect: {
    type: 'start_record' | 'stop_record' | 'recognize_voice' | 'none'
  }
}
```

### 12.4 `handleSubmit(payload)`

输入：

```js
{
  state: {},
  context: {},
  capability: {}
}
```

输出：

```js
{
  ok: true,
  submitPayload: {},
  nextState: {},
  successState: {}
}
```

### 12.5 `buildCloseDecision(payload)`

输入：

```js
{
  state: {},
  reason: 'cancel' | 'mask'
}
```

输出：

```js
{
  action: 'close_direct' | 'persist_draft' | 'confirm_discard',
  canPersistDraft: true,
  hasPendingChanges: true
}
```

## 13. 第一版必须保持兼容的旧字段

因为当前首页闪录逻辑已经存在，第一版改造时建议兼容以下旧概念：

- `quickEntryMode`
- `quickEntryForm`
- `quickEntrySelectedProjectId`
- `quickEntrySelectedProjectMeta`
- `quickEntryAiSummary`
- `quickEntryAiNextSuggestion`
- `quickEntryVoicePreviewText`
- `showQuickEntrySuccessPanel`

建议做法：

- 第一阶段先通过 adapter 映射旧结构到新结构
- 不要一开始就试图一刀切删除所有旧字段

## 14. 第一版不建议做的事情

以下做法容易把协议重新搞乱，第一版不建议采用：

- 给每个页面增加自己的 `quickEntryForm`
- 让组件内部直接调用云函数
- 让组件自己决定路由跳转
- 草稿按页面独立存多份
- 首页保留旧协议，其他页面使用另一套新协议

## 15. 验收标准

这份协议生效后，后续实现至少要满足：

1. 首页、项目页、外发页、我的页使用同一套组件接口
2. 组件不直接感知“当前是哪个页面的业务实现”
3. 录音、AI、草稿、项目匹配都通过共享控制层流转
4. 页面跳转决策不放进组件内部
5. 后续新增 `project_detail` 闪录入口时，无需重新设计一套字段体系

## 16. 建议下一步

有了这份协议后，下一步建议按下面顺序推进：

1. 先创建 `quick-entry-sheet` 组件空壳
2. 先把首页现有状态通过 adapter 喂给组件
3. 再把项目页、外发页、我的页接入同一组件
4. 再抽 `quick-entry-controller`
5. 最后清理首页旧实现

当前建议将本文档作为闪录宿主重构的字段蓝图执行。
