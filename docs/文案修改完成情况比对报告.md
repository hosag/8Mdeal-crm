## 四、修改模式分析

### 4.1 修改了哪些
- **小程序前端 `.wxml` 模板文案**：修改力度最大，约 85% 的可见文案已优化
- **页面级 `.js` 中的 data 初始化文案**：部分修改（如title、subtitle、empty文案）
- **配置型文案**：如 `STATUS_FILTERS`、`TYPE_FILTERS` 等数组中的label已大量修改

### 4.2 未修改哪些
- **工具函数层（`utils/`）**：`entitlement-guard.js`、`billing.js`、`notification-meta.js` 等核心文案配置基本未动
- **服务层（`services/`）**：`data.js`、`runtime.js`、`task-feedback.js` 等未动
- **云函数（`cloudfunctions/`）**：完全未改，所有AI提示语、错误消息、通知模板保持原样
- **admin-web**：不在本次修改范围（如确实未计划修改可忽略）

### 4.3 修改断层风险
存在明显的"**前端改了、后端没改**"断层：

**示例1 - "推进动作"vs"任务"**
- 前端页面：用户看到的已是"任务"、"待办任务"
- 云函数返回：AI生成的摘要、通知推送、研判结果中仍大量使用"推进动作"
→ 用户会在时间线、AI摘要、消息推送中继续看到"推进动作"

**示例2 - "只读"**
- 前端横幅：可能已改为"仅可查看"
- 后端报错：`services/runtime.js` 仍为"当前账号为只读状态，暂时无法继续保存"
→ 用户触发保存时弹出的toast仍是旧文案

**示例3 - AI错误提示**
- 前端AI sheet："当前无法生成项目研判"等完全未改
→ 与报告中建议的"生成失败，请重试"差距明显

**示例4 - "写入"**
- 多个核心页面的任务完成sheet仍写"提交后自动写入时间线"
→ 报告建议改为"保存"，但此处仍用"写入"
