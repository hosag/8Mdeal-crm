# 腾讯云正式商用 AI 接入方案

这份文档用于回答一个核心问题：

- 当前 CRM 已经接通 `cloudbase_default`
- 也已经接通 `openai_compatible` 通道
- 但正式商用时，到底该走哪条腾讯云方案，哪些套餐能用，哪些不能用

适用时间点：

- 当前内测阶段
- 即将进入收费版本准备阶段
- 后续上线正式商用阶段

相关延伸文档：

- 多模型路由、后台切换和未来用户模型模式开放，请参考 [AI多模型路由与用户切换方案](/Users/shaominhe/成交CRM-CodeX版/docs/AI多模型路由与用户切换方案.md)

## 一、先给结论

结论分成 3 条：

1. 不建议把腾讯云个人版 `Token Plan / Coding Plan` 用在本 CRM 的正式后端。
2. 当前联调和内测阶段，可以继续保留 `cloudbase_default` 与 `openai_compatible` 双通道。
3. 正式收费版本，优先使用“正式模型服务 / 正式 API Key / CloudBase 官方支持模型接入”，而不是个人版 AI 工具套餐。

这不是单纯的技术选择，而是合规性、稳定性、可运营性和可收费能力共同决定的。

## 二、为什么不建议用个人版 Token Plan / Coding Plan

根据腾讯官方当前文档，个人版 `Token Plan / Coding Plan` 主要面向 AI Coding 工具、个人开发和交互式使用场景，不适合本 CRM 这种业务后端：

- 小程序用户会通过云函数调用 AI
- 这是自定义业务后端
- 会有自动化调用和持续调用
- 后续还会进入正式收费和商用阶段

对本项目来说，个人版套餐有 4 个问题：

1. 场景不匹配  
   个人版套餐更偏向开发工具场景，不是标准的 SaaS 后端推理调用场景。

2. 商用风险高  
   你这个 CRM 明确是要收费的，后续还会涉及用户套餐、AI 额度、订阅和订单，必须尽量避免把非正式商用路径作为底层核心依赖。

3. 权益体系不好做  
   个人版套餐是面向账号本身，不适合映射到“每个 CRM 用户还有多少 AI 额度”这种产品化运营体系。

4. 迁移成本高  
   前期如果偷快用个人版套餐，后面改正式服务时，权限策略、成本策略、风控策略都要重走一轮。

## 三、当前系统里的 3 条 AI 通道分别是什么

### 1. `cloudbase_default`

当前默认接的是：

- `providerType = cloudbase`
- `cloudbaseProvider = hunyuan-exp`
- `defaultModel = hunyuan-turbos-latest`

这条通道的特点：

- 接入最简单
- 适合先把 AI 能力跑通
- 对当前小程序 + CloudBase 云函数架构最自然
- 适合作为保底通道和默认回退通道

适合阶段：

- 联调期
- 内测期
- 正式版早期的保底通道

### 2. `openai_compatible`

这条通道当前已经可以接：

- DeepSeek
- OpenAI 兼容网关
- 其他兼容 OpenAI 协议或 Responses 协议的供应商

这条通道的特点：

- 灵活
- 可快速切模型
- 适合比较模型质量、速度和成本
- 后续可扩展多供应商容灾

适合阶段：

- 联调期
- 模型选型期
- 正式商用后的扩展通道

### 3. `provider_plan`

这不是供应商通道，而是额度策略。

它表示：

- 本地不再以前置 `aiTokensRemaining` 阻断 AI 调用
- 真正的调用成本由上游供应商记账

它不等于：

- 自动接入腾讯个人版 `Token Plan`
- 自动绑定某个腾讯云套餐

## 四、你现在最适合的阶段性接法

### 阶段 A：当前联调与内测

建议：

- 默认保留 `cloudbase_default`
- 同时保留 `deepseek_primary`
- 额度策略优先可用 `provider_plan`

原因：

- 现在第一目标是把闪录、摘要、下一步建议链路跑稳定
- 此时“先跑通模型”比“先做严密本地额度扣减”更重要
- 出问题时可以在 `cloudbase_default` 和 `openai_compatible` 之间切换

推荐配置：

- 生产保底路由：`cloudbase_default`
- 对比验证路由：`deepseek_primary`
- `quotaPolicy`：联调可先用 `provider_plan`

### 阶段 B：收费前准备

建议：

- 保留双通道，但开始把主业务路由收敛
- 同时把本地权益层补齐
- 开始从 `provider_plan` 过渡到 `local_quota`

原因：

- 你已经明确要做套餐、订阅、额度包、试用期和只读限制
- 这些能力如果不上本地权益层，正式收费会很难运营

推荐配置：

- 闪录项目匹配：优先速度快的模型
- 跟进摘要：优先稳定、语言质量好的模型
- 下一步建议：优先业务表达更稳定的模型

此阶段重点不是彻底去掉 `provider_plan`，而是把它从“默认模式”降级成“联调和兜底模式”。

### 阶段 C：正式商用

建议：

- 用户权益控制切回 `local_quota`
- 上游模型使用正式商用 API 通道
- 保留第二供应商做故障切换

正式版推荐结构：

- 权限层：本地 `entitlements`
- 计费层：本地套餐 / 流量包 / 审计流水
- 推理层：腾讯云正式模型服务或正式 API Key
- 兜底层：第二供应商或第二路由

## 五、正式商用建议的腾讯云路径

如果你希望 AI 主通道继续优先留在腾讯云生态，建议优先按下面顺序考虑：

1. `cloudbase_default` 继续保留为腾讯云生态内默认通道  
   适合继续作为默认保底路径。

2. 使用腾讯云正式模型服务 / 正式 API 接入能力  
   这是后续更稳的正式商用方向，尤其适合：
   - 用户规模扩大
   - 需要稳定 SLA
   - 需要合规采购和正式成本核算

3. 在 AI 配置中心保留第二通道  
   例如：
   - `deepseek_primary`
   - 未来的正式 OpenAI 兼容供应商

这样做的好处：

- 主通道出问题时可快速切换
- 模型成本上涨时可做迁移
- 某一任务类型效果不好时可独立换模型

## 六、为什么正式版最终更推荐 `local_quota`

你的产品不是“单纯调用 AI 的工具”，而是“带套餐和订阅的 AI CRM”。

所以正式版真正需要的是：

- 用户还能不能继续用 AI
- 套餐包含多少 AI 额度
- 试用期什么时候结束
- 额度包是否到账
- 后台能不能人工补量
- 过期后是停用还是只读

这些能力必须由你本地系统掌控，而不是完全依赖云厂商套餐。

因此建议是：

- 联调期：`provider_plan`
- 收费期：`local_quota`
- 供应商侧真实成本仍然照常存在，但用户层的产品权益由本地控制

## 七、推荐的最终架构

建议你按下面结构长期演进：

1. 用户层  
   小程序前端只感知“是否可用、还剩多少权益、当前套餐是什么”。

2. 权益层  
   由本地 `accounts / entitlements / plans / subscriptions / usage ledger` 决定用户是否还能调用 AI。

3. 路由层  
   由 `featureFlags.ai_model_routing_v1` 决定哪个任务走哪个 provider。

4. 推理层  
   调用腾讯云正式模型服务、CloudBase 官方通道，或正式商用的 OpenAI 兼容供应商。

5. 成本层  
   供应商账单是真实成本来源，但不直接暴露给 CRM 用户。

## 八、当前最推荐的执行顺序

按现在项目进度，建议你这样推进：

1. 当前继续保留：
   - `cloudbase_default`
   - `deepseek_primary`

2. 联调和真机验证阶段：
   - 默认先用 `provider_plan`
   - 把三段 AI 路由调稳定

3. 支付与权益层联调完成后：
   - 逐步切回 `local_quota`
   - 把 AI 能力纳入正式套餐和额度包体系

4. 正式收费前：
   - 明确主通道
   - 明确备用通道
   - 完成至少一次“主通道故障 -> 备用通道切换”的演练

## 九、项目内的明确建议

对当前仓库，我建议如下：

- `cloudbase_default`：保留，作为腾讯云默认保底通道
- `deepseek_primary`：保留，作为当前高性价比可用通道
- `openai_primary`：保留，但仅作为扩展或兼容通道，不作为当前主路径
- `provider_plan`：保留，用于联调与过渡
- `local_quota`：作为正式收费版本的目标默认策略

## 十、最后一句判断

如果目标是“尽快做出可收费版本”，最稳妥的路径不是把腾讯个人版 `Token Plan / Coding Plan` 接进后端，而是：

- 短期用当前已通的正式可调用通道把功能跑稳
- 中期把本地权益系统立起来
- 正式版把 AI 调用绑定到你自己的套餐和额度体系上

这条路径虽然比“先蹭个人版套餐”慢一点，但能避免后面在收费、合规和系统迁移上付出更大代价。
