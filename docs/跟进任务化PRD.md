# 跟进任务化 PRD

本文档用于定义成交 CRM 小程序的“跟进任务化”能力。该能力的目标不是再做一个泛任务系统，而是把销售跟进从“记录发生了什么”，升级为“明确下一步要做什么、最晚什么时候做完、是否已经做完”。

本文档包含：

- 功能目标与业务边界
- 用户场景与任务类型
- 页面交互方案
- 数据模型与状态流转
- 云函数与页面改造清单
- 验收标准与分期计划

适用前提：

- 云环境已接通
- `projects`、`followUps` 已进入真实数据流
- 当前首页、我的项目、项目详情、外发项目已完成实用版收口
- [saveFollowUp](/Users/shaominhe/成交CRM-CodeX版/cloudfunctions/saveFollowUp/index.js) 已承担跟进记录写入能力

## 一、目标与原则

### 1.1 目标

当前项目已经能记录“发生了什么”，但仍然缺少“下一步要做什么”。跟进任务化的第一阶段目标是：

1. 把每次跟进后的下一步动作明确下来
2. 让动作具备清晰归属、截止时间和完成状态
3. 让首页、项目详情、消息提醒都围绕真实动作运转
4. 让 CRM 从“记录工具”变成“推进工具”

### 1.2 核心原则

- 跟进记录与待办任务必须分开建模，不能混在一条文本里
- 任务必须挂在具体项目下，不能做成脱离项目的通用 todo
- 任务必须归属于当前项目拥有者，不能做多人分派
- 任务状态要简单直接，优先可执行，不做复杂流转
- 第一阶段先做销售推进任务，不做审批流或跨部门工单
- “项目外发”和“信息分享”必须严格区分
- 信息分享接收方只有查看权限，不能创建、修改、完成任务

### 1.3 第一阶段范围

第一阶段只做“销售推进任务”，支持以下基础能力：

1. 创建任务
2. 明确任务归属
3. 设置截止时间
4. 修改状态
5. 在首页、项目详情、跟进页查看任务
6. 将逾期任务纳入首页待办与提醒体系

### 1.4 明确不做

第一阶段不做：

- 多级审批
- 子任务树
- 评论区 / 聊天式协作
- 跨项目任务
- 甘特图、日历视图
- 自动派单给企业微信 / 邮件
- 多人分配任务

## 二、问题定义

当前系统里，跟进后通常只记录：

- 跟进内容
- AI 摘要
- 下次跟进时间

但真实销售推进往往还需要更细的动作，例如：

- 待发方案
- 待报价
- 待回访
- 待签约
- 联系人信息未录入

这些动作如果只写在跟进摘要里，会出现 4 个问题：

1. 只有文字，没有明确任务归属
2. 只有“下次跟进时间”，没有动作截止时间
3. 首页只能提示“该跟进”，但不知道具体该做什么
4. 做没做完没有状态，时间一久就丢失

因此，必须把“跟进记录”和“待办动作”拆成两套能力：

- `followUps` 负责沉淀发生过什么
- `tasks` 负责推进接下来做什么

## 三、用户角色与场景

## 3.1 用户角色

第一阶段涉及 2 类业务身份：

1. 当前项目拥有者
   当前真正有操作权、负责推进项目的人

2. 外部接收方
   在“项目外发”模式下接手项目的新拥有者，或在“信息分享”模式下仅查看授权详情的人

## 3.2 核心场景

### 场景 A：跟进后明确下一步动作

用户刚结束一通电话或一次拜访，需要立刻把下一步动作落下来，而不是只写一句“后续继续推进”。

例如：

- 明天下午 3 点前发方案
- 周五前给客户报价
- 下周一回访采购经理

### 场景 B：项目详情里能看出当前推进卡点

用户进入项目详情，应该能一眼看到：

- 当前有哪些未完成任务
- 哪些任务已经逾期
- 最近完成了哪些任务

### 场景 C：首页显示今天真正要做的动作

首页不能只显示“哪个项目该跟进”，还应该能显示“今天要发方案、要报价、要回访”的具体动作。

### 场景 D：项目外发后，动作归属切换清晰

项目一旦通过“项目外发”被对方接手，新的项目拥有者应在自己的项目池中维护任务；原拥有者只做外发追踪，不再继续维护推进任务。

## 4.2 任务归属

### 功能名称

任务归属规则

### 规则

- 每个任务默认归属于当前项目拥有者
- 不支持把任务分配给其他账号
- 若当前项目是“我接手的项目”，任务归属人为接手后的当前拥有者
- “信息分享”模式下，接收方只有查看权限，不产生任务归属
- 已外发后的原项目侧不再继续新增推进任务

### 第一阶段实现建议

第一阶段直接固定为：

1. 当前项目拥有者 = 任务拥有者
2. 不出现任务分派 UI
3. 不出现负责人选择器

### 用户价值

- 避免任务责任模糊
- 符合私人 CRM 的单人使用定位

---

## 4.3 截止时间

### 功能名称

任务截止时间

### 规则

- 每个任务必须可设置截止日期和时间
- 截止时间不等于项目的 `nextFollowUpDate`
- 截止时间表示“这个动作最晚何时要完成”
- `nextFollowUpDate` 表示“项目下次整体跟进节点”

### 关键区别

示例：

- 任务：明天 15:00 前发方案
- 项目下次跟进：后天 10:00 回访客户确认反馈

这两个时间都要保留，不能二选一。

### 用户价值

- 把“动作截止”与“项目推进节奏”拆开
- 首页可以更准确地判断今天该做什么

---

## 4.4 任务状态

### 功能名称

任务状态流转

### 推荐状态

第一阶段建议使用 4 个状态：

1. `pending`
   待处理

2. `in_progress`
   进行中

3. `done`
   已完成

4. `canceled`
   已取消

### 派生状态

不单独存库，由系统计算：

- `overdue`
  当前时间晚于截止时间，且状态不是 `done` / `canceled`

### 状态规则

- 新建任务默认 `pending`
- 用户开始处理可切换 `in_progress`
- 完成后切换 `done`
- 若任务失效，可标记 `canceled`

### 用户价值

- 每个动作都有完成闭环
- 能统计项目当前真正还有多少动作没做

---

## 4.5 任务完成

### 功能名称

完成任务

### 完成方式

第一阶段建议提供两种方式：

1. 在项目详情页点击“完成”
2. 在任务详情弹层或任务卡上点击“完成”

### 完成时可补充的字段

- 完成时间
- 完成人
- 完成结果简述

### 第一阶段建议

完成时结果简述先做可选，不强制。

### 自动联动

任务完成后：

- 首页今日任务应自动消失或移入已完成
- 对应提醒自动关闭
- 项目详情任务区实时刷新

## 六、数据模型设计

## 6.1 为什么单独建集合

不建议把任务直接塞进 `followUps` 或 `projects`：

- `followUps` 是记录型数据，一次跟进可带多个任务
- `projects` 是主数据，不适合承载大量任务状态变更
- 任务需要独立更新状态、截止时间、完成结果

因此建议新增集合：

- `tasks`

## 6.2 `tasks` 集合建议结构

```js
{
  _id: '',
  projectId: '',
  sourceFollowUpId: '',
  ownerOpenid: '',
  ownerName: '',
  creatorOpenid: '',
  creatorName: '',
  title: '',
  type: 'send_solution | send_quote | callback | meeting | contract | collect_info | other',
  priority: 'high | normal | low',
  status: 'pending | in_progress | done | canceled',
  dueAt: new Date(),
  dueDateText: '2026-04-18 15:00',
  description: '',
  resultSummary: '',
  isSharedProjectTask: false,
  handoverStatusSnapshot: '',
  createdAt: new Date(),
  updatedAt: new Date(),
  startedAt: null,
  completedAt: null,
  completedByOpenid: '',
  completedByName: '',
  canceledAt: null,
  canceledByOpenid: '',
  canceledByName: ''
}
```

## 6.3 与现有字段的关系

### `nextFollowUpDate`

- 继续保留在 `projects`
- 表示项目整体下一次跟进节点

### `tasks.dueAt`

- 新增到 `tasks`
- 表示某个具体动作的截止时间

### `sourceFollowUpId`

- 用于追溯该任务是从哪条跟进记录里生成的

## 七、状态流转与业务规则

## 7.1 任务状态流转

推荐主流程：

`pending -> in_progress -> done`

补充流转：

- `pending -> canceled`
- `in_progress -> canceled`

### 明确不支持

第一阶段不做复杂回退流转，例如：

- `done -> pending`
- `done -> in_progress`

如确需重开，建议新建一条任务或单独做“重新打开”能力，放到后续版本。

## 7.2 逾期规则

逾期判断条件：

- 当前时间 > `dueAt`
- `status` 不是 `done`
- `status` 不是 `canceled`

## 7.3 已外发项目规则

### 分享方侧

- 项目一旦外发并完成管理权转移，原拥有者侧不允许继续新增推进任务
- 原拥有者只在“外发项目”中查看后续推进结果

### 接手方侧

- 接手后在自己的项目下创建与维护任务
- 任务完全属于新的项目拥有者推进链路

### 信息分享侧

- 信息分享接收方无法创建、编辑、完成任务
- 信息分享不会生成新的任务链路

## 7.4 任务与跟进的联动

### 跟进创建任务

- 一条跟进可以创建多条任务

### 跟进完成任务

第一阶段不自动把任务与新跟进强绑定，但允许用户手动完成任务。

### 后续可扩展

- 保存跟进时勾选“本次已完成任务”
- 自动关闭对应任务

## 十、与提醒能力的关系

跟进任务化与“提醒与消息”强相关，但不等价。

### 跟进任务化负责

- 定义要做什么
- 什么时候做完
- 是否完成

### 提醒与消息负责

- 告诉用户该处理哪个任务
- 告诉用户任务是否逾期
- 告诉用户项目外发后是否已被对方接手与推进

### 建议联动

后续提醒系统应优先围绕任务生成消息，而不是只围绕项目生成消息。

例如：

- `待发方案任务今天到期`
- `报价任务已逾期`
- `外发项目已有新的推进记录`

## 十一、验收标准

第一阶段完成后，至少满足以下 10 条：

1. 用户能在录入跟进时创建 1 到 3 条推进任务
2. 每条任务都有明确归属、截止时间、状态
3. 项目详情能看到当前未完成任务
4. 项目详情能看到逾期任务
5. 首页能感知今日到期任务和逾期任务
6. 我的项目页能看到每个项目的任务数量摘要
7. 任务完成后，页面状态实时更新
8. 已外发只读项目，分享方不能继续创建任务
9. 项目外发后的新拥有者可以在自己的项目下维护任务
10. 任务与跟进记录能通过 `sourceFollowUpId` 关联追溯

## 十三、开发建议顺序

建议按下面顺序落地，风险最低：

1. 先建 `tasks` 集合和云函数接口
2. 升级 `saveFollowUp`，支持跟进带任务写入
3. 在项目详情页先把任务读出来
4. 再把跟进页任务录入 UI 接上
5. 最后再把首页和我的项目做任务摘要联动

---

## 十四、产品判断标准

跟进任务化做成以后，判断是否真的有效，不看“任务数量多不多”，而看下面 4 件事：

1. 跟进后，用户能不能明确写下下一步动作
2. 两天后回来，用户能不能一眼知道哪些动作还没做
3. 首页能不能告诉用户今天具体先做什么
4. 项目外发后，任务责任和操作权是否清晰转移

如果这 4 件事做到了，这个功能才算真正让 CRM 从“记录工具”升级为“推进工具”。
