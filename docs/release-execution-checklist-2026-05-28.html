T0 提审门槛

T0 保持现有闪录策略

1. 隐私合规接入

目标：不再裸用录音/媒体相关隐私接口；补齐用户授权同步链路与后台声明，避免录音、媒体选择、上传链路在真机或审核环境被封禁。

核心文件范围：app.json、pages/index/index.js、pages/follow-up/follow-up.js、pages/projects/projects.js、pages/project-detail/project-detail.js、pages/tasks/tasks.js，建议新增统一工具如 utils/privacy-authorize.js。

  • 新增统一隐私授权工具：封装 wx.getPrivacySetting、wx.onNeedPrivacyAuthorization、agreePrivacyAuthorization 相关逻辑，避免每页各写一套。
  • 把录音、语音转写、媒体选择入口改成“先确认隐私同意，再走 authorize(scope.record) 或 chooseMedia”。
  • 首页闪录、跟进页、项目页、项目详情页、任务页这五条语音链路统一接入同一个隐私授权工具。
  • 微信后台补齐《用户隐私保护指引》：至少覆盖录音、媒体选择、文件上传涉及的信息收集类型。
  • 若要兼容手机号授权方案，预留 getPhoneNumber|agreePrivacyAuthorization 组合按钮能力。

验收： 未同意隐私时，点击录音/媒体入口不会直接报错；用户同意后功能继续；审核环境不再出现隐私声明缺失类报错。

T0

2. 缓存与本地草稿账号隔离

目标：杜绝同设备换号或身份切换后，A 账号缓存/草稿回显到 B 账号。

核心文件范围：utils/page-cache.js、utils/core-page-cache.js、pages/index/index.js、pages/projects/projects.js、pages/shared-out/shared-out.js、pages/project-detail/project-detail.js、pages/follow-up/follow-up.js、utils/home-entry-guide.js、app.js。

  • 新增统一账号命名空间工具，例如 getAccountScopedKey(baseKey)，统一基于 accountId 或稳定身份前缀生成 key。
  • pageMemoryCache 不再用裸 key，首页、项目列表、外发页、项目详情缓存全部改成账号级 key。
  • 本地 storage key 全部改成账号级：follow-up-draft、homeQuickEntryDraftsV1、homeQuickEntryProjectAliasesV1、homeEntryGuideSettings 以及相关语音提示标记。
  • 在 app.js 增加“账号上下文变更清理”逻辑：当 refreshAccount() 识别到账号变化或首次初始化时，清掉旧账号缓存和旧 storage。
  • 旧格式 key 不做迁移复用，直接视为废弃缓存，避免跨账号串读。

验收： A 账号打开过首页、项目详情、跟进草稿后切到 B 账号，B 不会先看到 A 的任何项目、联系人、AI 摘要、草稿内容。

T0

3. bindPhone 改为可信手机号验证

目标：不再接受前端手填手机号直接把 phoneVerified 置为 true；恢复手机号绑定作为真实信任边界，而不是伪门槛。

核心文件范围：pages/phone-bind/phone-bind.js、pages/phone-bind/phone-bind.wxml、services/data.js、cloudfunctions/bindPhone/index.js，并联动下单与写入门槛链路。

  • 前端从“手动输入手机号”切到微信官方手机号授权按钮方案；若短期无法完成，至少不要再把手输手机号写成 phoneVerified=true。
  • 云函数 bindPhone 改为接收可信授权凭证，不再接收裸 phoneNumber 作为验证依据。
  • 绑定成功后再刷新 resolveAccount / getEntitlements，保证写入能力、下单能力与真实绑定状态一致。
  • 回归检查所有 ACCOUNT_PHONE_REQUIRED 调用链：saveProject、saveFollowUp、flowProject、createBillingOrder、prepareBillingPayment。

验收： 未完成官方手机号授权时，账号仍是未绑定；完成授权后才变成已绑定，并能通过写入与支付门槛。

T0 不要误修成指定接收人模型

4. 外发接手 claim 原子化

目标：消除外发卡接手过程中的并发竞争、重复导入、后写覆盖和归属竞争。

核心文件范围：cloudfunctions/openSharedProject/index.js，必要时联动 cloudfunctions/createShareRecord/index.js 的字段设计。

  • 把“检查 shareRecord 是否已被锁定 / 项目是否已转交”与“写入 lock / 写入接手归属”做成原子流程。
  • 建议做法：先用事务 claim shareRecord 的接手锁，再继续导入项目与跟进；或者把最小可判定的 ownership lock 放在事务里 compare-and-set。
  • 避免多个接收方同时通过前置判断后，分别创建 imported project 或覆盖 handoverToOpenid。
  • 保持当前产品语义：现阶段外发卡仍是“持卡方可接手”，不要在没有产品改版的前提下硬塞 recipientAccountId 强校验。

验收： 并发模拟下，同一张外发卡最终只能有一个成功接手者；源项目归属与 shareRecord 锁定状态一致，不会出现双导入或后写覆盖。

不要误修的两件事

1. 不要把闪录重新改回入口前置权益校验

当前正确策略是：点击“闪录”先开半窗，后台静默校验，用户点击下一步或真正提交时再按动作类型拦截。开发不要把 openQuickEntrySheet() 前面重新塞回 ensureActionAllowed('quick_entry') 入口阻断。

2. 不要在未改产品模型前，硬把外发修成“指定接收人强校验”

当前产品不是定向转交模型。现阶段必须修的是原子性与并发 claim，不是强行要求 recipientAccountId 与当前用户相等。除非产品明确改版为定向接收卡，否则这会把正常分享/转交行为误伤掉。

提审前 smoke 清单

  • 未同意隐私时，首页闪录、跟进页录音、项目页录音、项目详情录音、任务页录音，都能先触发正确的隐私流程，而不是直接失败。
  • 同一设备 A 账号访问首页/项目详情/跟进草稿后切换 B 账号，B 账号首屏和详情不回显 A 的任何内容。
  • 手机号未授权时，无法通过“绑定完成”绕过写入与下单门槛；授权完成后能力恢复正常。
  • 同一张外发卡两端并发打开时，只允许一端成功接手，另一端收到稳定且可解释的反馈。
  • 外发页在缓存命中时也能同步通知读态；跟进、任务、成交后再进外发页状态不陈旧。
  • 分享卡、项目复盘、任务完成、转交流程等浮层出现时，自定义 tab bar 不会上浮挡按钮。

建议执行顺序

  1. 先做 T0-2 缓存账号隔离 和 T0-4 外发 claim 原子化。这两项最容易形成真实错数据或错归属。
  2. 并行做 T0-1 隐私合规接入 和 T0-3 手机号可信验证。这两项决定提审与后续支付链路可靠性。
  3. T0 合完后，立即补 T1 的 shared-out 脏标记、通知同步、tab bar 浮层隐藏、密钥 fallback 清理。
  4. 最后做一次完整 smoke，再收口 project.config.json 的发布配置与提审信息。
