## 一、联调目标

完成后，你应该能跑通这条链路：

1. 前端创建订单
2. 前端准备支付
3. 云函数向微信统一下单，拿到 `prepay_id`
4. 前端调用 `wx.requestPayment`
5. 微信支付官方异步通知回调到 `receiveWechatPaymentNotify`
6. 回调函数验签、解密、映射
7. 内部适配层推进到账 / 关闭 / 退款
8. 订单、订阅、流量包、审计日志都落库

## 五、必须准备的 feature flag

### 5.1 `billing_internal_operator_v1`

建议数据：

```json
{
  "flagKey": "billing_internal_operator_v1",
  "enabled": true,
  "payload": {
    "operatorKey": "请替换成高强度内部密钥",
    "operatorId": "billing_internal"
  },
  "updatedAt": "2026-04-28T10:00:00.000Z",
  "updatedBy": "system"
}
```

作用：

- 保护内部到账、关闭、退款、回调适配

### 5.2 `billing_payment_profile_v1`

建议数据：

```json
{
  "flagKey": "billing_payment_profile_v1",
  "enabled": true,
  "payload": {
    "profileCode": "wechat_jsapi_v1",
    "provider": "wechat_pay",
    "mode": "native_jsapi",
    "appId": "你的小程序AppID",
    "mchId": "你的商户号",
    "notifyFunctionName": "receiveWechatPaymentNotify",
    "merchantConfigReady": true,
    "clientPayload": {
      "timeStamp": "",
      "nonceStr": "",
      "package": "",
      "signType": "RSA",
      "paySign": ""
    }
  },
  "updatedAt": "2026-04-28T10:00:00.000Z",
  "updatedBy": "system"
}
```

当前代码口径：

- `mode = native_jsapi` 代表目标是走小程序 `wx.requestPayment`
- `notifyFunctionName` 建议直接写 `receiveWechatPaymentNotify`
- `merchantConfigReady = true` 只是允许系统尝试统一下单和签名前端参数
- 真正的敏感材料不应放进 `featureFlags`

## 六、必须配置的云函数环境变量

以下环境变量是当前代码真正会读取的。

### 6.1 统一下单与前端签名相关

- `BILLING_WECHAT_PAY_APPID`
- `BILLING_WECHAT_PAY_MCHID`
- `BILLING_WECHAT_PAY_MERCHANT_SERIAL_NO`
- `BILLING_WECHAT_PAY_PRIVATE_KEY`
- `BILLING_WECHAT_PAY_NOTIFY_FUNCTION`
- `BILLING_WECHAT_PAY_NOTIFY_URL`
- `BILLING_WECHAT_PAY_NOTIFY_BASE_URL`
- `BILLING_WECHAT_PAY_API_BASE`
- `BILLING_WECHAT_PAY_DEBUG_PREPAY_ID`

### 6.2 官方回调验签与解密相关

- `BILLING_WECHAT_PAY_PUBLIC_KEY`
- `BILLING_WECHAT_PAY_PUBLIC_KEY_ID`
- `BILLING_WECHAT_PAY_API_V3_KEY`

### 6.3 推荐的配置原则

- `BILLING_WECHAT_PAY_APPID`
  建议与小程序正式 `AppID` 一致
- `BILLING_WECHAT_PAY_MCHID`
  填真实商户号
- `BILLING_WECHAT_PAY_MERCHANT_SERIAL_NO`
  填商户 API 证书序列号
- `BILLING_WECHAT_PAY_PRIVATE_KEY`
  填 PEM 私钥全文
- `BILLING_WECHAT_PAY_NOTIFY_FUNCTION`
  建议填 `receiveWechatPaymentNotify`
- `BILLING_WECHAT_PAY_NOTIFY_URL`
  如果你已经有完整公网地址，就直接填它
- `BILLING_WECHAT_PAY_NOTIFY_BASE_URL`
  如果你的网关 URL 规则是“公共前缀 + 云函数名”，可以用它拼
- `BILLING_WECHAT_PAY_API_BASE`
  默认用 `https://api.mch.weixin.qq.com`
- `BILLING_WECHAT_PAY_DEBUG_PREPAY_ID`
  只用于内部冒烟，不建议正式环境长期保留
- `BILLING_WECHAT_PAY_PUBLIC_KEY`
  填微信支付公钥 PEM 全文
- `BILLING_WECHAT_PAY_PUBLIC_KEY_ID`
  填微信支付公钥 ID，例如 `PUB_KEY_ID_...`
- `BILLING_WECHAT_PAY_API_V3_KEY`
  必须是 32 位

### 6.4 PEM 和私钥填写注意

如果控制台环境变量不方便直接输入多行，可以：

- 先按带换行的原始 PEM 粘贴
- 如果平台只允许单行，再把换行转义成 `\n`

当前代码已经兼容 `\n` 转回真实换行。

## 七、回调 URL 应该怎么指

当前仓库的回调函数是：

- `receiveWechatPaymentNotify`

代码层预期：

- 微信支付异步通知 URL 最终命中这个函数
- 原始请求头必须透传：
  - `Wechatpay-Timestamp`
  - `Wechatpay-Nonce`
  - `Wechatpay-Signature`
  - `Wechatpay-Serial`
- 原始 body 不能被网关改写

如果你是自己做网关转发，必须特别注意：

- 不要把 body 先 parse 再 stringify 后转给函数
- 不要丢任何签名头
- 不要改字符编码

因为微信回调验签是基于“原始 body + 原始头”做的。

## 九、联调顺序

建议严格按下面顺序做。

### 9.1 验证账户与手机号

1. 打开小程序
2. 确认 `resolveAccount` 正常
3. 先完成手机号绑定

需要确认：

- 账户不是只读异常状态
- 付款前不会报 `ACCOUNT_PHONE_REQUIRED`

### 9.2 验证套餐与订单

1. 进入套餐页
2. 点击可购买商品
3. 进入订单详情页

需要确认：

- `orders` 有 `pending` 订单
- `orderId` 已生成
- `productCode` 正确

### 9.3 验证支付准备

1. 在订单详情页点“准备支付”
2. 看页面上的诊断信息

需要确认：

- `transactionId` 已生成
- `merchantTradeNo` 已生成
- `paymentSession.mode = native_jsapi`
- `paymentSession.merchantConfigReady = true`
- `paymentSession.privateKeyReady = true`

如果还没拿到真实 `prepay_id`，要重点看：

- `预支付单状态`
- `预支付单来源`
- `渠道下单时间`
- `渠道响应状态`
- `渠道下单结果`
- `待补配置项`

### 9.4 验证统一下单

如果统一下单成功，预期：

- 页面显示 `预支付单状态 = 已提供`
- 页面显示 `预支付单来源 = wechat_unified_order`
- `paymentSession.clientPayload` 已完整
- 主按钮从“重新准备支付”变成“尝试拉起支付”

如果统一下单失败，先不要急着看前端：

- 优先看订单页里的 `渠道下单结果`
- 再看云函数日志

### 9.5 验证前端拉起

1. 在真机环境中点击“尝试拉起支付”
2. 看微信支付收银台是否被调起

需要确认：

- 开发者工具环境不一定支持真实支付
- 优先用真机
- 拉起成功不等于到账成功
- 最终到账仍以异步回调为准

### 9.6 验证异步回调

如果用户完成支付，预期：

1. 微信支付异步通知打到 `receiveWechatPaymentNotify`
2. 函数验签成功
3. 解密成功
4. 内部转调 `handleBillingPaymentCallback`
5. 再转调 `markBillingOrderPaid`

最终你要看到：

- `orders.status = paid`
- `paymentTransactions.status = success`
- `billingCallbackEvents.status = applied`
- `adminAuditLogs` 有记录

### 9.7 验证权益到账

如果商品是订阅，预期：

- `subscriptions` 有新增或延期

如果商品是语音包 / AI 包，预期：

- `usageLedger` 有正向发放记录

## 十、订单页各字段怎么理解

为了后面排查更快，这里直接给你解释。

### `预支付单状态`

- `已提供`
  说明系统已经拿到 `prepay_id`
- `未提供`
  说明统一下单还没成功，或者你还在用占位模式

### `预支付单来源`

- `wechat_unified_order`
  说明真实统一下单成功
- `env.BILLING_WECHAT_PAY_DEBUG_PREPAY_ID`
  说明你在用调试占位
- `feature_flag.debugPrepayId`
  说明你在用配置调试值

### `签名策略`

- `server_rsa`
  说明服务端已经成功生成前端支付签名
- `none`
  说明当前还不能生成真实前端支付参数

### `渠道下单结果`

- `无`
  说明还没尝试，或尝试成功且错误被清空
- 有错误文本
  说明统一下单确实发起了，但微信返回了错误或网络异常

## 十一、常见错误与处理

### 报错：`merchantConfigReady`

说明：

- `billing_payment_profile_v1.payload.merchantConfigReady` 还是 `false`

处理：

1. 把 `merchantConfigReady` 改成 `true`
2. 重新准备支付

### 报错：`merchantPrivateKey`

说明：

- 商户私钥环境变量没配对

处理：

1. 检查 `BILLING_WECHAT_PAY_PRIVATE_KEY`
2. 确认 PEM 内容完整
3. 确认换行没丢

### 报错：`merchantSerialNo`

说明：

- 商户证书序列号没填

处理：

1. 检查 `BILLING_WECHAT_PAY_MERCHANT_SERIAL_NO`

### 报错：`notifyUrl`

说明：

- 统一下单必需的回调地址没形成

处理：

1. 直接配置 `BILLING_WECHAT_PAY_NOTIFY_URL`
2. 或者配置 `BILLING_WECHAT_PAY_NOTIFY_BASE_URL + BILLING_WECHAT_PAY_NOTIFY_FUNCTION`

### 订单页显示 `渠道下单结果 = WECHAT_PAY_UNIFIED_ORDER_TIMEOUT`

说明：

- 云函数发起统一下单超时

处理：

1. 检查云环境网络是否能访问微信支付
2. 检查商户配置是否正确
3. 重试一次并看云日志

### 订单页显示 `渠道下单结果 = WECHAT_PAY_UNIFIED_ORDER_FAILED:401...`

说明：

- 通常是签名、商户号、证书序列号、私钥不匹配

处理：

1. 检查 `mchid`
2. 检查商户私钥
3. 检查商户 API 证书序列号
4. 确认是不是用了错误商户环境

### 回调返回 `signature verify failed`

说明：

- 微信回调签名校验失败

处理：

1. 检查 `BILLING_WECHAT_PAY_PUBLIC_KEY`
2. 检查 `BILLING_WECHAT_PAY_PUBLIC_KEY_ID`
3. 检查网关有没有改写 body
4. 检查 `Wechatpay-*` 头有没有透传

### 回调返回 `decrypt failed`

说明：

- `APIv3 Key` 不正确，或回调密文被破坏

处理：

1. 检查 `BILLING_WECHAT_PAY_API_V3_KEY`
2. 确认是 32 位
3. 确认 body 原文没有被改写
