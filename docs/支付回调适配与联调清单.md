# 支付回调适配与联调清单

本文档用于说明当前成交 CRM 小程序收费底座里的“支付回调适配层”怎么部署、怎么联调，以及它和订单、到账、退款云函数之间的关系。

当前时间点的真实边界：

- 已有订单、支付准备、到账、关闭、失败、退款骨架
- 还没有接入真实微信支付拉起与官方回调
- `handleBillingPaymentCallback` 只是内部适配层
- 它的职责是把第三方支付回调状态翻译成系统内部订单状态推进

---

## 一、当前回调链路定位

当前支付相关云函数职责如下：

- `createBillingOrder`
  负责创建 `pending` 订单
- `prepareBillingPayment`
  负责生成或复用 `paymentTransactions` 待支付记录，以及标准化 `paymentSession`
- `handleBillingPaymentCallback`
  负责把外部回调状态翻译成内部动作
- `markBillingOrderPaid`
  负责订单到账、订阅发放、流量包发放、审计留痕
- `updateBillingOrderStatus`
  负责关闭、失败、退款，以及退款回滚权益

正确调用关系是：

1. 前端创建订单
2. 前端准备支付
3. 第三方支付平台回调适配层
4. 适配层调用内部到账或状态流转函数

也就是说，后续即使切到真实微信支付，也不应该让微信支付回调直接改数据库，而是继续先经过 `handleBillingPaymentCallback`。

## 二、已新增的适配层

建议部署路径：

- [cloudfunctions/handleBillingPaymentCallback/index.js](/Users/shaominhe/成交CRM-CodeX版/cloudfunctions/handleBillingPaymentCallback/index.js)

当前能力：

- 校验内部操作密钥 `operatorKey`
- 解析第三方状态
- 支持状态映射：
  - `success / paid / succeeded / trade_success` -> `mark_paid`
  - `fail / failed / error / payment_failed` -> `fail`
  - `close / closed / cancelled / trade_closed` -> `close`
  - `refund / refunded / refund_success` -> `refund`
  - `pending / processing / userpaying / notpay` -> 忽略，不推进状态
- 对同一个 `callbackTraceId` 做软幂等复用
- 写入 `billingCallbackEvents`
- 调用：
  - `markBillingOrderPaid`
  - `updateBillingOrderStatus`
- 记录 `adminAuditLogs`

补充说明：

- `prepareBillingPayment` 现在会生成内部 `transactionId`
- 同时写入 `merchantTradeNo`
- 并返回标准 `paymentSession`
- 当前 `paymentSession.mode = placeholder`
- 后续接真实微信支付时，只需要把 `clientPayload` 等真实参数填进去

## 五、必须准备的 feature flag

当前内部到账和回调适配都依赖：

- `billing_internal_operator_v1`
- `billing_payment_profile_v1`

建议文档结构：

```json
{
  "flagKey": "billing_internal_operator_v1",
  "enabled": true,
  "payload": {
    "operatorKey": "请替换成你自己的内部密钥",
    "operatorId": "billing_internal"
  }
}
```

注意：

- `operatorKey` 不能留空
- `handleBillingPaymentCallback`
- `markBillingOrderPaid`
- `updateBillingOrderStatus`

这 3 个内部函数当前都依赖它做保护，不应直接暴露给前端普通按钮。

`billing_payment_profile_v1` 建议结构：

```json
{
  "flagKey": "billing_payment_profile_v1",
  "enabled": false,
  "payload": {
    "profileCode": "wechat_jsapi_v1",
    "provider": "wechat_pay",
    "mode": "native_jsapi",
    "appId": "wx1234567890",
    "mchId": "1900000109",
    "notifyFunctionName": "handleBillingPaymentCallback",
    "merchantConfigReady": false,
    "clientPayload": {
      "timeStamp": "",
      "nonceStr": "",
      "package": "",
      "signType": "RSA",
      "paySign": ""
    }
  }
}
```

说明：

- `enabled = false` 时，系统仍返回 `placeholder` 或不可拉起会话
- `mode = native_jsapi` 代表走微信小程序 `wx.requestPayment`
- `merchantConfigReady` 只表示云端商户密钥与签名材料已经就绪
- 真正的商户私钥、证书序列号、APIv3 Key 不建议放在 `featureFlags`
- 这些敏感项后续应放在云函数环境变量或更安全的密钥管理位置

当前 `prepareBillingPayment` 还支持以下环境变量：

- `BILLING_WECHAT_PAY_APPID`
- `BILLING_WECHAT_PAY_MCHID`
- `BILLING_WECHAT_PAY_MERCHANT_SERIAL_NO`
- `BILLING_WECHAT_PAY_PRIVATE_KEY`
- `BILLING_WECHAT_PAY_NOTIFY_FUNCTION`
- `BILLING_WECHAT_PAY_NOTIFY_URL`
- `BILLING_WECHAT_PAY_NOTIFY_BASE_URL`
- `BILLING_WECHAT_PAY_API_BASE`
- `BILLING_WECHAT_PAY_DEBUG_PREPAY_ID`

微信支付官方回调入口还依赖：

- `BILLING_WECHAT_PAY_PUBLIC_KEY`
- `BILLING_WECHAT_PAY_PUBLIC_KEY_ID`
- `BILLING_WECHAT_PAY_API_V3_KEY`

当前用途：

- 若 `mode = native_jsapi`
- 且 `merchantConfigReady = true`
- 且存在 `appId + mchId + merchant serial + privateKey + notifyUrl`

系统会优先尝试向微信统一下单接口创建 `prepay_id`

如果已经拿到 `prepay_id`：

- 云函数会继续在服务端生成 `wx.requestPayment` 所需的 `clientPayload`

如果还没拿到 `prepay_id`：

- 可以临时用 `BILLING_WECHAT_PAY_DEBUG_PREPAY_ID` 冒烟验证前端拉起参数结构

注意：

- 这一步只解决“前端签名参数生成”
- 现在已经补上“统一下单请求骨架”，但仍依赖真实商户配置和可访问微信支付网络环境
- `BILLING_WECHAT_PAY_DEBUG_PREPAY_ID` 只适合内部冒烟验证，不适合正式商用

微信支付官方通知入口：

- [receiveWechatPaymentNotify/index.js](/Users/shaominhe/成交CRM-CodeX版/cloudfunctions/receiveWechatPaymentNotify/index.js)

当前能力：

- 校验 `Wechatpay-*` 签名头
- 用 `APIv3 Key` 解密回调资源
- 解析：
  - `TRANSACTION.SUCCESS`
  - `TRANSACTION.CLOSED`
  - `REFUND.SUCCESS`
- 转成内部事件后调用 `handleBillingPaymentCallback`
- 返回微信支付要求的 `{ code, message }` 回包结构

注意：

- `BILLING_WECHAT_PAY_PUBLIC_KEY` 建议放微信支付公钥 PEM 全文
- `BILLING_WECHAT_PAY_PUBLIC_KEY_ID` 用于校验微信支付公钥 ID
- `BILLING_WECHAT_PAY_API_V3_KEY` 必须是 32 位字符串

## 七、状态映射规则

当前内部映射固定如下：

| 第三方状态 | 内部动作 | 结果 |
| --- | --- | --- |
| `success` | `mark_paid` | 订单改 `paid`，并发放权益 |
| `fail` | `fail` | 订单改 `failed` |
| `close` | `close` | 订单改 `closed` |
| `refund` | `refund` | 订单改 `refunded`，并回滚权益 |
| `pending` | `ignore` | 不更新订单状态 |

后续如果接微信支付官方回调：

- 只需要在适配层里把微信字段翻译成以上内部状态
- 不应该重写到账、退款、订单状态机

## 八、手工联调顺序

建议严格按下面顺序验证。

### 8.1 先验证订单创建

1. 前端进入套餐页
2. 点击一个真实商品
3. 确认 `orders` 里生成 `pending` 订单

需要重点确认：

- `orderId`
- `accountId`
- `productCode`
- `status = pending`

### 8.2 再验证支付准备

1. 打开订单详情页
2. 点击“准备支付”
3. 确认 `paymentTransactions` 中生成 `pending` 记录

需要重点确认：

- `orderId`
- `accountId`
- `status = pending`
- `channel = wechat_pay`
- `transactionId` 已生成
- `merchantTradeNo` 已生成
- `paymentSession.mode = placeholder`
- `paymentSession.callbackFunctionName = handleBillingPaymentCallback`

如果你准备验证 JSAPI 模式，还需要额外确认：

- `paymentSession.readinessCode`
- `paymentSession.merchantConfigReady`
- `paymentSession.privateKeyReady`
- `paymentSession.prepayId`
- `paymentSession.prepayIdSource`
- `paymentSession.signStrategy`
- `paymentSession.channelOrder.requestAt`
- `paymentSession.channelOrder.requestError`
- `paymentSession.channelOrder.responseStatusCode`
- `paymentSession.missingConfigKeys`
- `paymentSession.clientPayload`

### 8.3 再验证成功回调

手工调用 `handleBillingPaymentCallback`：

```json
{
  "operatorKey": "你的内部密钥",
  "provider": "wechat_pay",
  "orderId": "ord_xxx",
  "status": "success",
  "externalTransactionId": "wx_txn_001",
  "callbackTraceId": "wx_success_001",
  "paidAt": "2026-04-28T10:00:00.000Z",
  "source": "manual_callback_test"
}
```

预期结果：

- `orders.status = paid`
- `paymentTransactions.status = success`
- 若是订阅商品：
  - `subscriptions` 有新增或延期
- 若是语音包 / AI 包：
  - `usageLedger` 有正向发放记录
- `billingCallbackEvents` 有 `applied` 记录
- `adminAuditLogs` 有审计记录

### 8.3.1 如果验证微信官方通知入口

需要额外确认：

- HTTP 入口最终映射到 `receiveWechatPaymentNotify`
- 请求头能透传：
  - `Wechatpay-Timestamp`
  - `Wechatpay-Nonce`
  - `Wechatpay-Signature`
  - `Wechatpay-Serial`
- 原始 body 不能被网关改写

成功时建议回包：

```json
{
  "code": "SUCCESS",
  "message": "success"
}
```

如果验签或解密失败，当前函数会返回：

```json
{
  "code": "FAIL",
  "message": "signature verify failed"
}
```

或：

```json
{
  "code": "FAIL",
  "message": "decrypt failed"
}
```

### 8.4 验证失败回调

使用新订单，手工调用：

```json
{
  "operatorKey": "你的内部密钥",
  "provider": "wechat_pay",
  "orderId": "ord_xxx",
  "status": "fail",
  "callbackTraceId": "wx_fail_001",
  "reason": "bank_rejected",
  "source": "manual_callback_test"
}
```

预期结果：

- `orders.status = failed`
- 对应 `paymentTransactions.status = failed`
- `failureReason` 被写入

### 8.5 验证关闭回调

使用新订单，手工调用：

```json
{
  "operatorKey": "你的内部密钥",
  "provider": "wechat_pay",
  "orderId": "ord_xxx",
  "status": "close",
  "callbackTraceId": "wx_close_001",
  "reason": "user_cancelled",
  "source": "manual_callback_test"
}
```

预期结果：

- `orders.status = closed`
- 对应 `paymentTransactions.status = failed`
- `closedReason = user_cancelled`

### 8.6 验证退款回调

先保证订单已成功到账，然后手工调用：

```json
{
  "operatorKey": "你的内部密钥",
  "provider": "wechat_pay",
  "orderId": "ord_xxx",
  "status": "refund",
  "callbackTraceId": "wx_refund_001",
  "reason": "manual_refund_test",
  "actionAt": "2026-04-28T10:30:00.000Z",
  "source": "manual_callback_test"
}
```

预期结果：

- `orders.status = refunded`
- `orders.fulfillmentStatus = reverted`
- 订阅则回滚订阅状态
- 流量包则写负向 `usageLedger`
- `billingCallbackEvents.status = applied`

## 九、幂等与唯一性说明

当前版本已经做了两层保护，但仍属于“骨架级”能力：

### 9.1 回调幂等

依赖：

- `billingCallbackEvents.provider`
- `billingCallbackEvents.callbackTraceId`

当前是软幂等：

- 如果同一个 `callbackTraceId` 已经处理完成，会直接复用结果
- 如果并发重复请求同时到达，仍可能双写事件日志
- 但订单到账与退款本身已尽量按订单维度做幂等，重复回调不应该重复发权益

### 9.2 交易号唯一性

当前还没有在数据库层面对 `paymentTransactions.transactionId` 做硬唯一约束。

后续真接微信支付前，建议至少补两件事：

1. 在 `paymentTransactions` 上增加“外部交易号 + 渠道”唯一校验策略
2. 在适配层把微信支付官方 `transaction_id` 和内部 `orderId` 同步落库

## 十、接入真实微信支付时怎么改

后续真正接微信支付时，建议只改这 3 个点：

1. `prepareBillingPayment`
   从“占位记录”升级为“签名并下发真实支付参数”
2. 新增或接入微信支付回调入口
   验签、解密、读取微信原始回调
3. 在 `handleBillingPaymentCallback`
   里把微信字段翻译成内部 `success / fail / close / refund / pending`

不建议改动：

- `markBillingOrderPaid`
- `updateBillingOrderStatus`
- 权益发放与退款回滚主逻辑

当前仓库状态下，这第 2 步已经有实现骨架：

- `receiveWechatPaymentNotify`

但还需要你在云端或网关层真正把微信支付回调 URL 指到这个函数。

这样可以把“支付渠道接入”与“业务发放逻辑”隔离开，后续即使切支付宝、Stripe 或别的渠道，也仍然复用内部状态机。

## 十一、当前前端支付按钮行为

订单详情页当前按钮逻辑如下：

- `placeholder` 模式：
  - 只生成支付会话
  - 不会调用 `wx.requestPayment`
- `native_jsapi` 但配置不完整：
  - 页面会展示待补配置项
  - 不会调用 `wx.requestPayment`
- `native_jsapi` 且 `canInvokePayment = true`：
  - 页面会尝试调用 `wx.requestPayment`
  - 但最终到账仍以回调适配层结果为准

因此，前端支付按钮已经具备未来可复用入口，但当前默认仍是“安全不可拉起”状态。
