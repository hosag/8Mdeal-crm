# 闪录公共组件改造方案

## 1. 背景

当前闪录能力完整挂载在首页 [pages/index/index.js](/Users/shaominhe/成交CRM-CodeX版/pages/index/index.js) 内部。

这带来两个明显问题：

- 从项目页、外发页、我的页点击底部 `闪录` 时，需要先跳到首页，再打开闪录半窗
- 用户关闭闪录后，即使逻辑上能回到来源页，体感上仍然像“切了两次页面”，交互偏重

现阶段的折中方案虽然能缓解“被留在首页”的问题，但没有解决根因：

- 闪录宿主仍然是首页
- 其他页面没有原地承载闪录的能力
- 页面切换成本仍然存在

因此，下一阶段应把闪录从“首页内功能”升级为“全局可复用的公共能力”。

## 2. 改造目标

本次改造的目标不是简单抽一层 UI，而是完成以下三件事：

1. 把闪录半窗抽成公共组件，支持首页、项目页、外发页、我的页原地打开
2. 把闪录核心流程从首页页脚本中剥离，沉淀为共享控制层
3. 统一闪录的上下文输入、关闭行为、草稿策略和成功回调，避免不同页面各写一套

改造完成后，目标体验应为：

- 用户在任意支持页面点击 `闪录`，都直接在当前页原地打开半窗
- 关闭、保存草稿、放弃本次闪录时，不发生页面跳转
- 保存成功后，根据来源场景执行一致、可预测的回调

## 3. 不在本次范围内的内容

为了避免一次改造过大，本次不包含以下事项：

- 不重做闪录产品流程本身
- 不重写语音识别、AI 理解、项目匹配云函数逻辑
- 不新增 H5 / Web 版本闪录
- 不顺带重做首页整体结构
- 不在本轮引入新的状态管理框架

本次关注点仅限于“宿主重构”和“交互去跳页化”。

## 4. 当前问题拆解

### 4.1 架构问题

当前闪录的 UI、状态、录音、AI、草稿、关闭逻辑全部集中在首页：

- UI 宿主：首页
- 入口：多个页面的底部导航
- 承载方式：其他页面跳转到首页后打开闪录

这说明闪录现在是“首页功能被全局借用”，而不是“全局公共能力”。

### 4.2 体验问题

当前路径存在以下体验损耗：

- 打开闪录时有明显页面切换感
- 关闭闪录时用户会感知一次返回
- 来源页面的上下文虽然可以恢复，但过程显得偏重
- 用户容易觉得自己被系统“带走了”

### 4.3 维护问题

如果继续在当前结构上打补丁，后续会出现以下风险：

- 首页脚本继续膨胀
- 闪录相关行为越来越依赖首页生命周期
- 新页面若也要接入闪录，只能继续复制入口跳转逻辑
- 后续想做“项目页默认锁定当前项目”等场景化能力时，代码会更绕

## 5. 改造原则

### 5.1 先拆宿主，再拆逻辑

先解决“在哪里承载闪录”的问题，再解决“谁负责闪录逻辑”的问题。

原因：

- 先让所有页面具备原地打开能力，可以先快速改善用户体验
- 再逐步把首页中的逻辑迁出，风险更可控

### 5.2 不是只抽 UI，而是组件加控制层一起抽

如果只抽出 WXML / WXSS，逻辑仍散落在各页面，会造成：

- 逻辑重复
- 回归点变多
- 后续难以统一修复问题

因此必须采用：

- 公共组件：负责界面、半窗、动画、局部交互
- 共享控制层：负责录音、识别、AI、草稿、项目匹配、提交流程

### 5.3 保持现有业务链路稳定

改造期间应优先复用现有云函数、数据结构和提交流程，避免架构重构时叠加业务重构。

### 5.4 优先保证“退出与草稿”行为一致

闪录当前最敏感的不是 UI，而是：

- 关闭
- 保存草稿
- 放弃本次闪录
- 成功后继续录入

这些行为必须在所有页面一致，否则会放大用户困惑。

## 6. 目标架构

建议拆成以下两层。

### 6.1 公共展示层

新增公共组件，建议命名为：

- `components/quick-entry-sheet`

职责：

- 承载闪录半窗
- 管理显隐、遮罩、过渡动画
- 渲染语音录入、识别结果、AI 理解、项目关联、动作区
- 抛出标准事件给页面或控制层

不负责：

- 云函数调用
- 草稿存储策略
- 项目匹配算法
- 提交成功后的业务跳转决策

### 6.2 共享控制层

新增共享逻辑层，建议命名为：

- `services/quick-entry-controller.js`

必要时再拆分为：

- `services/quick-entry-controller.js`
- `services/quick-entry-draft.js`
- `services/quick-entry-audio.js`
- `services/quick-entry-project-match.js`

职责：

- 初始化闪录状态
- 管理录音状态机
- 处理语音识别上传
- 调用 AI 摘要与下一步建议
- 执行项目候选匹配与纠正学习
- 管理草稿读写
- 统一关闭策略
- 统一保存提交
- 返回标准化结果给页面

### 6.3 页面接入层

首页、项目页、外发页、我的页只保留轻量接入代码：

- 打开闪录
- 传入来源上下文
- 接收成功回调
- 决定成功后是否继续当前页动作

页面不再直接持有整套闪录业务逻辑。

## 7. 页面上下文设计

为了支持不同页面原地打开，闪录需要接受统一的上下文输入。

建议上下文字段：

```js
{
  sourcePage: 'home' | 'projects' | 'project_detail' | 'shared_out' | 'mine',
  sourceScene: 'global_entry' | 'current_project' | 'shared_record',
  projectId: '',
  projectName: '',
  lockProject: false,
  readonlyProject: false
}
```

建议规则：

- 首页打开：`lockProject = false`
- 项目页打开：默认带当前 `projectId`，并优先锁定当前项目
- 项目详情页打开：默认带当前 `projectId`，并优先锁定当前项目
- 外发页打开：默认不锁定外发只读项目，只作为来源信息提示
- 我的页打开：空上下文打开

这样后续可以自然演进出更强的场景化行为，而不是把判断硬编码在首页里。

## 8. 草稿策略设计

这是本次改造的高风险点，必须先定规则。

建议采用“全局一份主草稿，按模式分桶，不按页面分桶”的策略。

建议原因：

- 闪录本质是用户级快速记录入口，不应因为来源页不同就生成多份互相冲突的草稿
- 用户从项目页进入一半，切到我的页继续，是合理场景
- 如果按页面分草稿，后续更容易出现“多个未完成草稿互相打架”

建议结构：

- `follow_up` 草稿一份
- `task` 草稿一份
- `project` 草稿一份

同时补一层来源信息：

- `lastSourcePage`
- `lastProjectId`
- `updatedAt`

这样可以在恢复草稿时给出合理提示，但不把草稿切碎。

## 9. 关闭与成功回调统一协议

建议对组件外暴露统一结果事件，而不是每个页面自己猜状态。

### 9.1 关闭结果

建议统一返回：

```js
{
  type: 'close',
  reason: 'cancel' | 'persist_draft' | 'discard' | 'mask' | 'success_close'
}
```

### 9.2 提交成功结果

建议统一返回：

```js
{
  type: 'success',
  mode: 'follow_up' | 'task' | 'project',
  projectId: '',
  continueProjectId: '',
  continueFollowUpMethod: '',
  createdRecordId: ''
}
```

页面接到结果后再决定：

- 关闭半窗，停留当前页
- 打开项目详情
- 继续录下一条

## 10. 分阶段实施计划

建议分四个阶段推进。

### 阶段 A：方案与边界冻结

目标：

- 明确组件边界
- 明确控制层边界
- 明确上下文协议、草稿策略、成功回调协议

产出：

- 本方案文档确认
- 字段协议确认
- 回归清单确认

### 阶段 B：公共宿主组件落地

目标：

- 新建 `quick-entry-sheet` 公共组件
- 让首页、项目页、外发页、我的页都能原地打开同一套半窗

此阶段允许：

- 控制逻辑暂时仍部分依赖首页现有函数

验收重点：

- 不再发生页面跳转
- 关闭后页面停留原地
- 半窗动画、键盘、遮罩行为正常

### 阶段 C：共享控制层抽离

目标：

- 把录音、识别、AI、草稿、项目匹配、提交逻辑迁出首页
- 页面只保留宿主与回调

验收重点：

- 首页不再是唯一逻辑宿主
- 项目页与首页行为保持一致
- 草稿策略统一

### 阶段 D：清理兼容代码与回归

目标：

- 删除中间过渡逻辑
- 清理基于首页跳转的旧入口
- 完成全链路真机回归

验收重点：

- 各入口行为一致
- 不再依赖 `openQuickEntry=1` 跳首页承载
- 旧兼容逻辑清理完成

## 11. 主要风险点

### 11.1 录音状态机迁移风险

当前录音、识别、暂停、关闭之间存在复杂状态关系。抽离时如果处理不当，容易出现：

- 录音关闭后未停止
- 识别中关闭导致状态残留
- 重新打开时按钮状态异常

### 11.2 页面生命周期差异风险

之前很多逻辑绑定在首页生命周期：

- `onLoad`
- `onShow`
- `onHide`

改为公共组件后，需要重新梳理：

- 半窗显隐时该做什么
- 页面隐藏时是否自动存草稿
- 页面切后台时是否停止录音

### 11.3 草稿冲突风险

如果草稿策略不统一，容易出现：

- 不同页面恢复出不同版本
- 放弃后仍被旧页面恢复
- 项目关联信息残留到下一次录入

### 11.4 成功后落点分叉风险

保存成功后，可能出现多种后续动作：

- 留在当前页
- 继续录下一条
- 打开刚保存的项目详情

如果不做统一协议，会导致每个页面单独写一套分支。

### 11.5 回归范围扩大风险

本次不是简单样式改动，而是入口层级重构。回归范围至少包括：

- 首页闪录
- 项目页闪录
- 外发页闪录
- 我的页闪录
- 草稿保存与恢复
- 放弃本次闪录
- 语音录入链路
- AI 项目匹配与下一步建议

## 12. 回归重点清单

正式开发时，至少覆盖以下场景：

1. 首页原地打开闪录，关闭后停留首页
2. 项目页原地打开闪录，关闭后停留项目页
3. 外发页原地打开闪录，关闭后停留外发页
4. 我的页原地打开闪录，关闭后停留我的页
5. 录音中关闭，录音能正确停止
6. 识别中关闭，结果不会残留污染下次录入
7. 保存草稿后，从其他页面打开仍能继续
8. 放弃本次闪录后，重新打开绝对空白
9. 项目页上下文打开时，默认关联当前项目
10. 成功保存后继续录下一条，状态能正确重置

## 13. 建议的实施顺序

为了降低风险，建议按下面顺序执行：

1. 先冻结本文档，不急着改代码
2. 先补一份“闪录公共组件字段协议”小文档或直接写进任务清单
3. 先做公共宿主组件最小版，只承接当前 UI
4. 再逐步把控制逻辑从首页迁出
5. 最后删除旧的跳首页承载逻辑

不建议的做法：

- 一次性把首页闪录全部搬走
- 一边改产品流程，一边改宿主结构
- 页面各自接一套闪录逻辑

## 14. 结论

这次改造的核心不是“把首页弹窗搬家”，而是把闪录从“首页功能”升级为“全局公共能力”。

推荐路线：

- 短期目标：所有入口原地打开，不再跳首页
- 中期目标：闪录逻辑去首页化，沉淀为共享控制层
- 长期收益：更轻的交互、更清晰的维护边界、更容易继续扩展到项目详情等场景

当前建议按本文档作为后续开发蓝图执行。
