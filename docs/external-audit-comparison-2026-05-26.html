外部审计报告对比评估

评估时间：2026-05-26。结论基于当前仓库代码快照、关键调用链核对，以及微信官方文档对隐私合规和 project.config.json 字段说明的交叉验证。我不建议把外部报告原样当作发布门槛清单；其中有几项判断方向是对的，但严重级别、根因归因、修复路径并不完全准确，尤其是 P0-1、P0-3、P0-5、P1-1、P2-7。

我的总判断

Confirmed Blockers
3
我明确支持作为上线前阻断项的有 3 类：隐私合规缺口、跨账号缓存泄漏、手机号绑定伪验证。
Conditional Blockers
1
外发接手链路存在原子性缺口，我认为必须修，但外部报告给的“指定接收人校验”不是当前产品形态下的正确修法。
Overstated Items
5+
至少 5 项被外部报告说重了，或者修复建议偏了方向，不应机械照抄执行。

是否应该现在直接提审

我的结论是：不建议在修完最核心的 4 个问题前提审，但这个“4 个问题”不是外部报告原样的 5 个 P0。更准确的提审门槛应是：

  • 补齐隐私合规接入与后台声明，不再裸用录音/媒体相关隐私接口。
  • 给页面缓存和本地草稿加账号命名空间，并补全账号切换/初始化时的清理机制。
  • 把 bindPhone 从“前端自填即验证通过”改成真实手机号授权或等价可信校验。
  • 把外发接手流程改成原子 claim，至少消除并发抢占和双写竞争。

P0-1 部分赞成 上线阻断 方向我赞成，修法我不赞成。当前代码确实调用了 wx.getRecorderManager、wx.chooseMedia、wx.cloud.uploadFile，但项目里没有任何 wx.getPrivacySetting、wx.onNeedPrivacyAuthorization、agreePrivacyAuthorization 接入，也没有看到隐私授权联动逻辑；app.json 里也没有历史上的 __usePrivacyCheck__ 配置。微信官方文档明确说明：未在后台《小程序用户隐私保护指引》中声明对应隐私类型，相关接口会被直接禁用，且会出现 api scope is not declared in the privacy agreement / appid privacy api banned 类错误。

但外部报告把它说成“必须在 app.json 增加 privacy 块、requiredPrivateInfos、permission.scope.record / scope.album / scope.writePhotosAlbum”并不准确。官方文档并没有把这些字段当成当前小程序隐私合规的必需修法，尤其 scope.record / scope.album 不是当前 app.json 权限描述的标准修法。 		先做两件事：
1. 在微信后台补齐《用户隐私保护指引》声明。
2. 前端补隐私授权接入，至少在录音/媒体入口前支持 wx.getPrivacySetting + agreePrivacyAuthorization 或被动监听授权事件。

参考核对：app.json，pages/index/index.js，pages/follow-up/follow-up.js，微信官方隐私文档。
P0-2 赞成 上线阻断 这是我明确同意的高风险问题，而且实际情况比报告写得还更糟一点。utils/page-cache.js 的 key 完全没有账号命名空间；restoreHomePageCache、restoreProjectsPageCache、restoreProjectDetailCache 会在账号上下文确认前优先回显旧缓存。只要出现同设备换微信号、应用内身份变化、或旧缓存残留，新账号就有机会先看到��账号的客户、跟进、AI 结果和外发信息。

另外，报告里说“有 clearPageCache 可调用”还算轻描淡写。当前 clearPageCache 只能删单 key，不是清空整个 store；而且代码里几乎没有任何账号切换、身份切换时的总清理入口。

本地持久化草稿也确实没有命名空间，比如 follow-up-draft:<projectId>、homeQuickEntryDraftsV1、homeQuickEntryProjectAliasesV1、homeEntryGuideSettings。 		必须做账号隔离：
1. page cache key 加 accountId 或 openid 前缀。
2. 本地草稿、项目别名、引导设置等 storage key 全部加账号命名空间。
3. 增加统一的“账号上下文切换/初始化清理函数”，清掉旧缓存和旧 storage。

参考核对：utils/page-cache.js，utils/core-page-cache.js，pages/index/index.js，pages/follow-up/follow-up.js。
P0-3 部分赞成 条件阻断 我同意这里有必须收口的接手安全问题，但我不同意外部报告给的核心定性。

当前产品和文案本身并没有“指定接收人”这个模型。无论 pages/share-config/share-config.js、pages/projects/projects.wxml、docs/项目复制与转交实施方案.md，表达的都是“把卡片发给要接手的人”，而不是“系统只允许预设某个人接手”。因此，要求现在去强制校验 shareRecord.recipientAccountId 是否等于调用方，这会和现有产品形态直接冲突。

但外部报告指出的另一个点是对的：cloudfunctions/openSharedProject/index.js 先读 shareRecord，后写 shareRecord 和 projects，整个 claim 过程没有事务 compare-and-set。两个接收方并发打开时，确实可能同时通过前置判断，再发生最后写入者覆盖、重复导入、归属竞争的问题。这个必须修。

所以我的结论是：不是“缺收件人校验”这个原因导致 P0，而是“claim 不是原子操作”这件事足够危险。如果产品后续要支持指定接收人，再引入 recipientAccountId 才是正确方向。 		当前产品形态下的正确修法：
1. 对 share claim 做事务化或单文档 CAS。
2. claim 成功前，不要提前写源项目归属。
3. 明确一张外发卡是否允许“任何持卡人先到先得”；若不允许，另加一次性 token 或显式接收人字段。

参考核对：cloudfunctions/openSharedProject/index.js:1058-1263，cloudfunctions/createShareRecord/index.js，相关产品文档与页面文案。
P0-4 赞成 上线阻断 这是实锤。cloudfunctions/bindPhone/index.js 直接接收前端传入的 phoneNumber，只做正则校验，然后把 accounts.phoneVerified 写成 true。前端页面 pages/phone-bind/phone-bind.js 也明确是“手动输入手机号”。

但系统其他地方已经把 phoneVerified 当成真实门槛：写入能力、套餐下单、支付准备都依赖它。也就是说，任何客户端都能自报手机号并绕过“先绑定再写入 / 再下单”的约束。这不是体验问题，是信任边界完全放错了。 		上线前必须改成可信校验链路。最现实的是：
1. 用 getPhoneNumber 授权码换取真实号码。
2. 绑定动作同时走隐私授权联动。
3. 在切换前，临时关闭“手机号绑定后可写 / 可下单”的强门槛，避免伪验证继续放行。

参考核对：cloudfunctions/bindPhone/index.js，pages/phone-bind/phone-bind.js，cloudfunctions/createBillingOrder/index.js，cloudfunctions/prepareBillingPayment/index.js。
P0-5 不同意 应修但非阻断 我不同意把这项定为 P0，也不同意其中一部分表述。project.config.json 目前确实带着 demo 信息，且 uploadWithSourceMap: true、minified: false、urlCheck: false 都不适合正式提审前的默认配置。微信官方文档也明确了这些字段的含义：minified 是上传时是否压缩，uploadWithSourceMap 是上传时是否携带 sourcemap，urlCheck 是是否检查安全域名和 TLS。

但把 urlCheck: false 解释成“任何漏配外域名都会被悄悄发布”是错的。它影响的是开发者工具上传前校验，不是运行时安全域名机制本身。运行时是否可请求，仍然受平台域名白名单约束。

所以这项更像提审前的发布卫生项，不是客户数据泄漏级别的上线阻断。 		建议提审前顺手改，但不必把它和 P0 安全漏洞混为一谈：
1. 改掉 demo 名称和描述。
2. 关闭 uploadWithSourceMap。
3. 开启 minified。
4. 若团队流程允许，打开 urlCheck 作为发布前防呆。

参考核对：project.config.json，微信官方 project config 文档。

我和外部审计的关键分歧

1. 隐私合规问题是真的，但不能按错文档字段去修

外部报告把“后台隐私声明 + 用户授权同步”问题，写成了“往 app.json 里堆 privacy 字段”的问题。这个方向会把团队带偏。正确重心是：后台声明、授权同步、接口调用前置，不是机械补几个可疑字段。

2. 外发接手问题的根因是 claim 原子性，不是现阶段缺了指定接收人

当前产品没有“指定接收人”能力，审计建议直接按 recipientAccountId 强校验，会和现有分享/转交流程冲突。必须坚持正确方向：先修并发 claim 和 ownership 更新原子性，再决定是否升级成真正的定向转交模型。

3. 闪录先弹半窗后校验，是当前已经确认过的产品策略

外部报告建议把 quick_entry 入口重新加前置拦截，这和你当前已经拍板的交互策略相反。这个点不应该被安全审计反向推翻。

4. project.config.json 需要收口，但不应被当成和手机号伪验证同等级的 P0

它更像发布卫生和源码暴露面控制，而不是直接的数据越权漏洞。要修，但别错误排序。

我建议的执行顺序

第一组：提审前必须完成

  • 隐私合规接入：后台声明 + 前端隐私授权同步链路。
  • 缓存与本地草稿账号隔离：page cache、flash draft、follow-up draft、guide settings 全部命名空间化。
  • bindPhone 改成可信手机号验证，不再接受前端自报即 phoneVerified=true。
  • 外发 claim 改成事务或 compare-and-set，避免并发抢占与归属双写。

第二组：建议和本轮缓存修复一起做

  • includeSharedOut: true 补齐到相关 mutator。
  • 外发页缓存命中分支补通知同步。
  • tab bar 浮层隐藏机制统一梳理，不只补 sync，还要补 hide 规则。
  • CONTACT_CRYPTO_SECRET 去掉硬编码 fallback，env 缺失直接报错。

第三组：后续优化，不阻断当前主线

  • tab-bar-navigation 请求版本号化。
  • getUserPreferences 去重请求。
  • saveUserPreferences 改成字段级 patch，避免互相覆盖。
  • 清理生产环境 console.info。
  • 给推荐码绑定增加频控。

最终结论

如果只回答一句话：我不支持把这份外部审计报告原样当作上线标准。它抓到了几个真正危险的问题，但也混入了若干“问题方向对、修法不对”以及“严重性放大”的项。

我会坚持的正确判断是：

  • 隐私合规、跨账号缓存泄漏、手机号伪验证，这三类必须正视，不能带着上线。
  • 外发 claim 必须加原子性保护，但不要在当前产品没设计“指定接收人”的情况下，生搬硬套接收人强校验。
  • 闪录先弹半窗、下一步再校验权益，这个交互方向可以坚持，不应该被审计意见推翻。
  • project.config.json 要收口，但不应该抬到和真实越权漏洞同级的 P0。
