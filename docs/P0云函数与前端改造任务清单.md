## 四、云函数任务清单

## 4.1 新增 `resolveAccount`

建议路径：

- `cloudfunctions/resolveAccount/index.js`

职责：

- 根据 `wxContext.OPENID` 查找 `accountIdentities`
- 若已存在，返回对应 `accountId`
- 若不存在，创建：
  - `accounts`
  - `accountIdentities`
  - 必要时补 `users.accountId`
- 初始化试用起止时间
- 返回最小账户摘要

返回建议：

```json
{
  "ok": true,
  "accountId": "acc_xxx",
  "status": "trialing",
  "phoneVerified": false,
  "trialEndsAt": "2026-05-05T09:00:00.000Z"
}
```

实施要点：

- 必须幂等
- 必须允许重复调用
- 不能因为 `users` 尚未完整存在而失败

## 4.2 新增 `getEntitlements`

建议路径：

- `cloudfunctions/getEntitlements/index.js`

职责：

- 读取 `accounts`
- 读取 `featureFlags`
- 读取 `subscriptions`
- 聚合 `projects` 当前数量
- 聚合 `usageLedger` 用量
- 计算并返回统一权益快照
- 必要时同步刷新 `entitlements`

返回建议至少包含：

- `status`
- `bindRequiredForWrite`
- `canCreateProject`
- `canSaveFollowUp`
- `canUseQuickEntry`
- `canUseSpeechToText`
- `canUseAi`
- `canShareOut`
- `projectLimit`
- `currentProjectCount`
- `voiceSecondsRemaining`
- `aiTokensRemaining`
- `reasonSummary`

实施要点：

- 前端和后端统一以这个结果为准
- 即使权益计算失败，也要返回明确错误，而不是静默放过

## 4.3 新增 `bindPhone`

## 4.4 改造 `login`

现状：

- 当前 `login` 主要是以 `_openid` 写入 `users`

建议改造目标：

- 不再把 `login` 当成最终账户入口
- 它只保留“补齐用户档案”的轻职责
- 正式账户初始化交给 `resolveAccount`

建议改法：

- 保持兼容，不立刻删除
- 在 `login` 中补写 `users.accountId`
- 若后续确认 `resolveAccount` 完全稳定，再考虑淡化 `login`

## 八、接口与错误码建议

## 8.1 P0 建议统一错误码

| 错误码 | 含义 | 典型场景 |
|------|------|------|
| `ACCOUNT_PHONE_REQUIRED` | 需要绑定手机号 | 首次正式保存、开通付费前 |
| `ENTITLEMENT_WRITE_DISABLED` | 当前为只读状态 | 试用已结束且未付费 |
| `ENTITLEMENT_PROJECT_LIMIT_REACHED` | 已达项目上限 | 免费项目数已满 |
| `ENTITLEMENT_SPEECH_EXHAUSTED` | 语音额度不足 | 录音转写前 |
| `ENTITLEMENT_AI_EXHAUSTED` | AI 额度不足 | AI 匹配或摘要前 |
| `ENTITLEMENT_SHARE_OUT_DISABLED` | 不允许外发 | 项目外发前 |
| `ACCOUNT_DISABLED` | 账户被禁用 | 风控或后台关闭 |

## 8.2 前端统一处理建议

每个错误码建议对应：

- 短提示文案
- 一个主要 CTA
- 一个次要 CTA

示例：

- `ACCOUNT_PHONE_REQUIRED`
  - 主 CTA：去绑定手机号
  - 次 CTA：先取消

- `ENTITLEMENT_PROJECT_LIMIT_REACHED`
  - 主 CTA：查看套餐
  - 次 CTA：稍后再说

## 十二、实施风险与注意事项

### 12.1 不要做的事

- 不要一开始就把所有云函数都重写为只认 `accountId`
- 不要直接删除 `_openid`
- 不要把支付逻辑提前混进 P0

### 12.2 容易踩坑的点

- 启动阶段先后顺序错误，导致页面先渲染再拿不到权益
- 手机号绑定设计过早耦合真机能力，阻塞架构推进
- 前端拦截了，但后端没拦截，导致可绕过
- 后端拦截了，但前端没解释清楚，导致用户感知为故障

### 12.3 P0 成功标准

P0 成功，不是“页面变多了”，而是：

- 系统已经知道“你是谁”
- 系统已经知道“你现在能做什么”
- 核心入口已经能一致地执行这套规则

做到这一步，P1 才能顺利接支付。
