# 项目复制与转交实施方案

## 一、产品目标

当前项目需要支持两类独立业务动作：

1. **复制项目**

   用户基于自己已有项目快速创建一个新项目，复制项目静态信息，后续仍由自己跟进。

2. **转交项目**

   用户把项目或客户新需求交给别人处理。转交时只保留两个清晰选项：

   - **全部转交**：对方接手原项目，自己只读查看。
   - **新建转交**：为对方创建一个新项目，只带客户基础信息和联系方式，自己继续管理当前项目。

这两类动作虽然底层可以复用同一套克隆/流转引擎，但在 UI 上必须保持两个明确入口，避免用户混淆。

## 二、核心原则

1. **复制后项目独立**

   复制出来的新项目与原项目没有任何自动同步关系。

2. **转交后项目独立**

   新建转交创建出的对方项目与当前项目没有任何后续同步。

3. **不做双向分支**

   不支持原项目更新同步到副本，也不支持副本更新回写原项目。

4. **不复制动态内容**

   复制项目和新建转交都不复制跟进记录、推进任务、任务完成历史、消息提醒、分享历史、成交记录、收益记录。

5. **只保留轻量来源关系**

   新项目可以记录来源项目 ID、来源项目名、来源模式，但这只是审计和提示，不是运行时绑定关系。

## 三、功能边界

### 3.1 复制项目

用户入口：

- 我的项目卡片更多操作
- 项目详情页更多操作
- 已成交、已流失项目也允许复制

用户行为：

```text
点击复制项目 -> 系统复制静态信息 -> 创建新项目 -> 打开新项目编辑页
```

不弹复制方式选择。

复制后打开：

```text
/pages/project-form/project-form?projectId=新项目ID&mode=edit&source=clone
```

或使用现有项目编辑页参数约定。

建议复制后 toast：

```text
已复制为新项目，可继续编辑
```

### 3.2 转交项目

用户入口：

- 项目详情页转交入口
- 我的项目卡片分享/转交入口

用户行为：

```text
点击转交项目 -> 选择转交方式 -> 确认 -> 生成分享/转交流程
```

转交方式只保留两个：

1. **全部转交**

   文案：

   ```text
   对方接手原项目，你保留只读查看。
   ```

2. **新建转交**

   文案：

   ```text
   为对方新建一个项目，只带客户和联系方式，你继续管理当前项目。
   ```

新建转交需要项目名称输入框。

默认项目名建议：

```text
客户名 · 新需求
```

如果客户名为空，兜底：

```text
新需求项目
```

确认按钮：

- 全部转交：

  ```text
  确认转交
  ```

- 新建转交：

  ```text
  新建转交
  ```

底部提示：

```text
新建转交不会同步当前项目历史，双方后续独立推进。
```

## 四、模式定义

底层统一使用三个模式，但 UI 只暴露必要动作。

```js
flowMode:
  transfer_original // 全部转交
  clone_static      // 复制项目
  clone_seed        // 新建转交
```

对应关系：

| UI 入口 | 用户动作 | flowMode | 目标归属 |
| --- | --- | --- | --- |
| 复制项目 | 复制项目 | clone_static | 当前用户 |
| 转交项目 | 全部转交 | transfer_original | 接收方接管原项目 |
| 转交项目 | 新建转交 | clone_seed | 接收方获得新项目 |

## 五、字段复制规则

### 5.1 clone_static：复制项目

用途：

```text
我基于当前项目创建一个新项目，仍由我自己跟进。
```

复制字段建议：

项目基础字段：

```js
projectName
clientName
description
estimatedAmount
expectedCommission
stage
tags
voiceAliases
followUpSilenceDays
contacts
privacyTags / shareTags 中与项目静态相关的字段
```

建议处理规则：

- 新项目 `projectName` 默认：

  ```text
  原项目名 · 复制
  ```

- 新项目 `stage` 默认：

  ```text
  线索
  ```

  理由：复制项目通常代表新需求、新项目线，用户进入编辑页后可手动调整。

- `estimatedAmount`、`expectedCommission` 可以复制，用户进入编辑页后自行修改或清空。
- `followUpSilenceDays` 可以复制，因为它是项目级偏好，不是动态任务。
- `contacts` 复制为新数组，不复用旧对象引用。

不复制字段：

```js
_id
_openid
accountId // 按新项目归属重新写
createdAt
updatedAt
isClosed
actualAmount
handoverStatus
handoverToOpenid
handoverToName
handoverAt
sharedFromOpenid
sharedFromName
sourceShareRecordId
sourceProjectId // 源关系字段要重新按本次复制写
nextFollowUpDate
aiReview
aiWakeResult
```

不复制集合数据：

```text
followUps
tasks
notifications
shareRecords
deals
earnings / payments / commissions
projectAssets / attachments 第一版不复制
```

来源字段写入：

```js
sourceProjectId: 原项目ID
sourceProjectName: 原项目名
sourceFlowMode: 'clone_static'
sourceFlowIntent: 'self_clone'
sourceFlowRecordId: 流转记录ID
sourceFlowCreatedAt: now
```

### 5.2 clone_seed：新建转交

用途：

```text
为接收方创建一个全新项目，只带客户基础信息和联系方式。
```

复制字段建议：

```js
projectName // 来自用户输入
clientName
contacts
voiceAliases // 可选，建议第一版不复制，避免对方语音匹配混乱
tags // 只复制基础客户标签，若当前标签混有项目阶段标签，可第一版不复制
description // 不复制，或写入一句来源说明
stage: '线索'
estimatedAmount: 0
expectedCommission: 0
followUpSilenceDays: 0
```

建议第一版最稳字段：

```js
projectName
clientName
contacts
stage: '线索'
description: `来自「原项目名」的新建转交`
```

不复制字段：

```text
原项目阶段
原项目金额
原项目描述
跟进记录
推进任务
附件
AI复盘
成交记录
收益记录
分享历史
消息提醒
```

来源字段写入：

```js
sourceProjectId: 原项目ID
sourceProjectName: 原项目名
sourceFlowMode: 'clone_seed'
sourceFlowIntent: 'transfer'
sourceFlowRecordId: 流转记录ID
sourceFlowCreatedAt: now
transferredFromOpenid: 发起人openid
transferredFromName: 发起人显示名
```

### 5.3 transfer_original：全部转交

沿用当前逻辑。

行为：

```text
原项目被接收方接手，发起方保留只读查看。
```

现有字段继续使用或补充：

```js
handoverStatus: 'handed_over'
handoverToOpenid
handoverToName
handoverAt
```

流转记录写入：

```js
flowMode: 'transfer_original'
flowIntent: 'transfer'
sourceProjectId: 原项目ID
targetOpenid
targetName
```

## 六、数据表设计

建议新增集合：

```text
projectFlows
```

用于统一记录复制、转交、新建转交。

字段建议：

```js
{
  _openid: 发起人openid,
  accountId: 发起人accountId,

  flowIntent: 'self_clone' | 'transfer',
  flowMode: 'transfer_original' | 'clone_static' | 'clone_seed',

  sourceProjectId,
  sourceProjectName,
  sourceProjectStage,
  sourceClientName,

  targetOpenid,
  targetAccountId,
  targetName,

  resultProjectId,        // clone_static / clone_seed 新项目ID
  resultShareRecordId,    // 如果仍需生成分享卡片
  resultStatus: 'created' | 'pending_accept' | 'accepted' | 'failed',

  copiedFieldSet: [],
  skippedFieldSet: [],

  note,
  createdAt,
  updatedAt
}
```

说明：

- `projectFlows` 是审计记录，不参与同步。
- 不做任何定时同步或反向更新。

## 七、云函数设计

建议新增统一云函数：

```text
flowProject
```

负责三种模式：

```js
{
  sourceProjectId: '',
  flowIntent: 'self_clone' | 'transfer',
  flowMode: 'transfer_original' | 'clone_static' | 'clone_seed',
  targetOpenid: '',
  targetAccountId: '',
  targetName: '',
  newProjectName: '',
  options: {
    openEditorAfterCreate: true
  }
}
```

返回：

```js
{
  ok: true,
  flowId,
  flowMode,
  resultProjectId,
  resultProjectName,
  nextUrl,
  message
}
```

### 7.1 clone_static 逻辑

1. 校验项目存在且属于当前用户可操作范围。
2. 校验当前账号有写权限、项目数量未超限。
3. 创建 `projectFlows` 记录。
4. 复制静态字段生成新项目。
5. 写入来源字段。
6. 更新 `projectFlows.resultProjectId`。
7. 返回新项目编辑页 URL。

返回示例：

```js
{
  ok: true,
  flowMode: 'clone_static',
  resultProjectId: 'xxx',
  nextUrl: '/pages/project-form/project-form?projectId=xxx&mode=edit&source=clone',
  message: '已复制为新项目'
}
```

### 7.2 clone_seed 逻辑

1. 校验源项目存在。
2. 校验接收方身份。
3. 校验发起方有转交/分享权限。
4. 创建 `projectFlows` 记录。
5. 为接收方创建新项目。
6. 只写客户和联系人基础信息。
7. 生成必要的接收通知或分享记录。
8. 返回转交结果。

这里要结合现有外发机制决定：

- 如果当前转交依赖分享卡片接收，则 `clone_seed` 先生成一个新建转交分享记录，对方打开后创建项目。
- 如果已有接收方 openid / accountId，则可以直接创建到对方项目池。

第一版建议沿用现有分享卡片机制，降低接收方识别难度。

### 7.3 transfer_original 逻辑

沿用现有转交云函数或迁入 `flowProject`。

如果迁入统一函数，旧入口可内部调用：

```js
flowProject({ flowMode: 'transfer_original' })
```

## 十、权益校验

复制项目：

- 需要创建项目权限
- 需要项目数量额度
- 如果只复制静态信息，不消耗 AI / 语音额度

新建转交：

- 发起方需要转交/分享权限
- 接收方创建项目时如何计入额度，需要沿用现有外发接收逻辑
- 不消耗 AI / 语音额度

全部转交：

- 沿用当前转交权限校验

## 十一、兼容与迁移

1. 当前已有全部转交流程保留，不破坏。
2. `shareRecords` 继续服务现有资料分享和外发卡片。
3. 新增 `projectFlows` 只记录新动作，不强制迁移旧数据。
4. 老项目没有来源字段时正常展示。
5. 新复制项目只显示轻提示，不影响主流程。

## 十二、实施顺序建议

第一步：底层能力

- 新增 `projectFlows` 集合使用约定。
- 新增 `flowProject` 云函数。
- 实现 `clone_static`。
- 实现 `clone_seed`。
- `transfer_original` 暂时可以继续用现有逻辑，后续再统一迁入。

第二步：我的项目复制

- 我的项目卡片增加“复制项目”入口。
- 项目详情增加“复制项目”入口。
- 调用 `flowProject clone_static`。
- 成功后跳项目编辑页。

第三步：转交面板收敛

- 转交项目面板改成两个选项：
  - 全部转交
  - 新建转交
- 新建转交展示项目名输入框。
- 新建转交调用 `flowProject clone_seed` 或现有分享卡片扩展逻辑。
- 全部转交继续走现有方式。

第四步：来源展示

- 新项目详情页轻提示：

  ```text
  复制自「原项目名」
  ```

  或：

  ```text
  来自「原项目名」的新建转交
  ```

- 我的项目卡片上可先不展示，避免卡片变重。

第五步：测试

需要覆盖：

- 进行中项目复制
- 已成交项目复制
- 已流失项目复制
- 复制后编辑项目名
- 复制后没有跟进记录
- 复制后没有推进任务
- 全部转交仍保持当前只读逻辑
- 新建转交只带客户和联系方式
- 新建转交不影响原项目
- 双方后续操作互不影响

## 十三、最终确认版产品结构

最终用户可见结构：

```text
复制项目
- 复制静态信息
- 创建为我的新项目
- 打开编辑页修改
```

```text
转交项目
- 全部转交
- 新建转交
```

最终底层模式：

```text
clone_static      我的项目复制
clone_seed        新建转交
transfer_original 全部转交
```

最终边界：

```text
不自动同步
不双向分支
不复制任务
不复制跟进记录
不复制收益和成交记录
只保留来源提示
```
