# P0 账户与权益数据表详细设计

本文档用于承接 [账户权益支付后台规划蓝图](/Users/shaominhe/成交CRM-CodeX版/docs/账户权益支付后台规划蓝图.md) 的 `P0` 阶段，把“账户、身份、手机号绑定、试用、免费、订阅前权益判断”落成可实施的数据结构方案。

适用范围：

- 微信小程序当前版本
- 个人私密 AI CRM
- 当前不做团队协同
- 当前不做主账号 / 子账号
- 当前不做正式 `unionid` 统一身份落地，但预留字段

---

## 一、P0 目标

`P0` 阶段只解决 5 件事情：

1. 为每个真实用户建立稳定的 `accountId`
2. 把微信身份、手机号、用户档案拆分清楚
3. 为试用、免费、付费、到期只读建立统一状态
4. 为语音与 AI 用量建立可扣减、可追查的数据结构
5. 为现有业务集合从 `_openid` 迁移到 `accountId` 做过渡准备

---

## 二、设计原则

### 2.1 拆层原则

不要把所有信息继续堆在 `users` 集合里。

建议拆成四层：

- `accounts`：账户主状态
- `accountIdentities`：身份映射
- `users`：档案与偏好
- `entitlements`：当前权益快照

### 2.2 迁移原则

P0 阶段不做激进替换，采用“新增字段 + 双写双读”的保守迁移方式。

### 2.3 对账原则

凡是会影响收费与用量的状态，都应该：

- 有主表
- 有流水
- 有更新时间
- 能解释当前值是怎么来的

---

## 三、命名与通用约定

### 3.1 ID 约定

- `accountId`：平台内部账户主键
- `planCode`：套餐编码
- `orderId`：订单号
- `traceId`：一次扣费或发放动作的追踪号

建议格式：

- `accountId`: `acc_` 前缀
- `orderId`: `ord_` 前缀
- `traceId`: `trc_` 前缀

### 3.2 时间字段约定

统一使用以下命名：

- `createdAt`
- `updatedAt`
- `startedAt`
- `expiresAt`
- `acceptedAt`
- `occurredAt`

### 3.3 布尔字段约定

统一用 `canXxx`、`isXxx`、`hasXxx` 风格：

- `canCreateProject`
- `isPrimary`
- `hasBoundPhone`

### 3.4 枚举字段约定

枚举值尽量使用英文稳定码，不直接把中文文案写入核心判断字段。

## 四、新增集合设计

## 4.1 `accounts`

用途：

- 平台账户主表
- 承载账号状态、试用周期、手机号绑定状态
- 作为支付、权益、订单的主归属对象

建议文档示例：

```json
{
  "accountId": "acc_20260428_0001",
  "status": "trialing",
  "phone": "",
  "phoneVerified": false,
  "primaryIdentityType": "wechat_mp",
  "trialStartedAt": "2026-04-28T09:00:00.000Z",
  "trialEndsAt": "2026-05-05T09:00:00.000Z",
  "currentAccessLevel": "trial_full",
  "lastActiveAt": "2026-04-28T09:00:00.000Z",
  "disabledReason": "",
  "createdAt": "2026-04-28T09:00:00.000Z",
  "updatedAt": "2026-04-28T09:00:00.000Z"
}
```

字段建议：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `accountId` | string | 是 | 平台账户主键 |
| `status` | string | 是 | `trialing / free_limited / active_paid / expired_readonly / disabled` |
| `phone` | string | 否 | 已绑定手机号，建议脱敏展示时另行处理 |
| `phoneVerified` | boolean | 是 | 手机号是否完成授权绑定 |
| `primaryIdentityType` | string | 是 | 当前主接入身份，第一版固定 `wechat_mp` |
| `trialStartedAt` | date | 否 | 试用开始时间 |
| `trialEndsAt` | date | 否 | 试用结束时间 |
| `currentAccessLevel` | string | 是 | `trial_full / free_readonly / paid_active / paid_readonly / disabled` |
| `lastActiveAt` | date | 否 | 最近活跃时间 |
| `disabledReason` | string | 否 | 被禁用时记录原因 |
| `createdAt` | date | 是 | 创建时间 |
| `updatedAt` | date | 是 | 更新时间 |

索引建议：

- 唯一索引：`accountId`
- 普通索引：`status + updatedAt`
- 普通索引：`phone`

## 4.6 `usageLedger`

用途：

- 记录语音与 AI 的用量流水
- 支撑扣减、赠送、补偿、审计和成本统计

建议文档示例：

```json
{
  "accountId": "acc_20260428_0001",
  "usageType": "voice_seconds",
  "sourceType": "speech_to_text",
  "sourceId": "quick_entry_20260510_0001",
  "delta": -38,
  "unit": "second",
  "beforeBalance": 3118,
  "afterBalance": 3080,
  "traceId": "trc_20260510_0001",
  "meta": {
    "projectId": "p123",
    "pageKey": "pages/index/index"
  },
  "occurredAt": "2026-05-10T09:00:00.000Z"
}
```

字段建议：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `accountId` | string | 是 | 账户主键 |
| `usageType` | string | 是 | `voice_seconds / ai_tokens` |
| `sourceType` | string | 是 | `speech_to_text / quick_entry_match / summarize_followup / grant / compensate` |
| `sourceId` | string | 否 | 来源业务对象 ID |
| `delta` | number | 是 | 正数为补发，负数为扣减 |
| `unit` | string | 是 | `second / token` |
| `beforeBalance` | number | 是 | 扣减前余额 |
| `afterBalance` | number | 是 | 扣减后余额 |
| `traceId` | string | 是 | 幂等追踪号 |
| `meta` | object | 否 | 附加上下文 |
| `occurredAt` | date | 是 | 发生时间 |

索引建议：

- 普通索引：`accountId + occurredAt`
- 普通索引：`traceId`
- 普通索引：`usageType + occurredAt`

注意：

- 每一次语音与 AI 消耗都必须有流水
- 流水必须支持幂等，避免重试时重复扣减

## 五、现有集合的增量字段设计

P0 不要求现有业务集合立即彻底重构，但必须先补齐 `accountId` 相关字段。

## 5.1 `users`

当前 `users` 已存在，建议改造成“用户档案与偏好表”。

新增字段建议：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `accountId` | string | 否 | 关联平台账户 |
| `phoneMasked` | string | 否 | 脱敏手机号 |
| `bindStatus` | string | 否 | `unbound / bound` |
| `agreementAcceptedVersion` | object | 否 | 各协议版本记录 |

保留原字段：

- `_openid`
- `nickName`
- `avatarUrl`
- `shareTags`
- `reminderSettings`
- `appearanceSettings`

说明：

- `users` 继续服务于小程序表现层
- `accounts` 才是收费、权益、后台的主账

## 5.2 `projects`

新增字段建议：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `accountId` | string | 否 | 归属账户 |
| `ownerAccountId` | string | 否 | 当前 owner |
| `sharedFromAccountId` | string | 否 | 外发来源账户 |
| `writeSource` | string | 否 | `legacy_openid / account_migrated` |

说明：

- 当前 `_openid` 先保留
- 后续 owner 判定逐步切到 `ownerAccountId`

## 5.3 `followUps`

## 七、试用与免费规则的落地字段

建议把规则拆成“全局配置 + 账户实际状态”两部分。

## 7.1 全局配置

存于 `featureFlags` 的 `trial_policy_v1`：

```json
{
  "trialDays": 7,
  "freeProjectLimit": 3,
  "trialVoiceSeconds": 600,
  "trialAiTokens": 50000,
  "readonlyAfterTrial": true,
  "writeRequiresPhoneBinding": true
}
```

## 7.2 账户实际状态

存于：

- `accounts`
- `entitlements`
- `usageLedger`

判断原则：

1. 根据 `trialEndsAt` 判断是否仍在试用内
2. 根据 `subscriptions` 判断是否有有效订阅
3. 根据 `usageLedger` 计算剩余语音与 AI 额度
4. 根据 `projects` 当前数与免费门槛判断是否可新增

## 九、P0 阶段迁移策略

## 9.1 第一阶段写入策略

新建或修改关键业务数据时：

- 继续保留 `_openid`
- 尽量补写 `accountId`

## 9.2 第一阶段查询策略

查询时优先级建议：

1. 有 `accountId` 时优先按 `accountId`
2. 无 `accountId` 时回退 `_openid`

## 9.3 第一阶段禁止动作

P0 阶段不建议：

- 删除现有 `_openid`
- 直接重写所有旧数据
- 一次性替换所有业务云函数查询逻辑

---

## 十、推荐初始化顺序

1. 建 `accounts`
2. 建 `accountIdentities`
3. 给 `users` 补 `accountId`
4. 建 `entitlements`
5. 建 `featureFlags`
6. 建 `usageLedger`
7. 给 `projects / followUps / tasks / shareRecords / deals / notifications` 增量补 `accountId`
8. 再准备 `plans / subscriptions / orders / paymentTransactions`

---

## 十一、P0 验收标准

1. 新用户首次进入后，系统能生成或解析稳定 `accountId`
2. `users`、`accounts`、`accountIdentities` 三层关系清晰
3. 系统能返回统一 `entitlements` 快照
4. 语音与 AI 扣减结构已具备，但 P0 可以先不接真实支付
5. 现有业务集合已具备 `accountId` 迁移字段，不阻塞后续演进

---

## 十二、实施备注

这份文档是 `P0` 数据结构稿，不代表这一轮要把所有集合全部开发完。

第一批真正需要优先落地的是：

- `accounts`
- `accountIdentities`
- `entitlements`
- `featureFlags`
- `users` 的增量字段

第二批再接：

- `usageLedger`
- `plans`
- `subscriptions`
- `orders`
- `paymentTransactions`

这样可以保持开发节奏：先把身份和规则站稳，再接支付和后台。
