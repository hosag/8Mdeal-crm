# 闪录中段低保真页面草图说明

本文档用于承接：

- [闪录中段流程重排方案](/Users/shaominhe/成交CRM-CodeX版/docs/闪录中段流程重排方案.md)

目标是把“录音后到保存前”的四个状态，进一步落成可执行的低保真页面说明，方便后续：

- 调整信息架构
- 先做页面重排
- 再做视觉收敛

这份文档不追求视觉细节，而是明确：

- 每一屏先看什么
- 模块从上到下怎么排
- 主按钮叫什么
- 哪些内容必须后置或折叠

## 1. 总体规则

后续四个状态统一遵守以下规则：

1. 顶部结构固定
2. 主内容区只服务当前阶段主任务
3. 底部固定动作区最多 1 个主按钮 + 1 到 2 个次动作
4. 除当前阶段核心内容外，其余能力尽量后置

统一顶部建议：

- 左侧：关闭
- 中间：阶段标题
- 右侧：空，或仅保留极轻状态文字

统一底部建议：

- 一个主按钮
- 一个次按钮或文本链接
- 不要出现 3 个以上同级动作

统一容器建议：

- 全页最多 1 个主容器 + 1 个辅助容器
- 不要堆很多同级卡片
- 输入区永远比解释区重

## 2. 整体状态流

建议按下面顺序推进：

1. 状态 A：录音 / 输入态
2. 状态 B：内容确认态
3. 状态 C：项目确认态
4. 状态 D：结果确认态

切换逻辑建议：

- A 到 B：录音结束，或手动输入达到最小长度
- B 到 C：用户确认内容可继续分析，或系统自动开始匹配项目
- C 到 D：用户确认项目，或系统高置信度自动锁定项目

## 3. 状态 A：录音 / 输入态

### 3.1 页面目标

只做一件事：

- 把这次情况记下来

### 3.2 低保真结构

```text
┌──────────────────────────┐
│ 关闭          闪录        │
├──────────────────────────┤
│                          │
│       [ 大录音按钮 ]      │
│                          │
│   正在记录你刚才说的话     │
│                          │
│  实时文字 / 输入中的文本    │
│  最多显示 2 到 4 行         │
│                          │
│  改为手动输入              │
│                          │
├──────────────────────────┤
│ 次动作         主按钮      │
└──────────────────────────┘
```

### 3.3 模块顺序

从上到下只保留：

1. 顶部栏
2. 录音主控区
3. 实时文本区
4. `改为手动输入`
5. 底部动作区

### 3.4 主次按钮建议

主按钮文案：

- 初始录音前：`开始录音`
- 录音中：`结束录音`
- 手动输入态：`继续`

次动作建议：

- `取消`

### 3.5 这一屏不该出现什么

不要出现：

- 项目候选
- AI 摘要
- 下一步建议
- 建议时间
- 阶段建议
- 风险提示
- 大段帮助说明

### 3.6 手动输入子态

若点击 `改为手动输入`，不切到复杂表单，而是同屏替换录音主控区。

建议结构：

```text
┌──────────────────────────┐
│ 关闭          闪录        │
├──────────────────────────┤
│ 这次发生了什么？          │
│ ┌──────────────────────┐ │
│ │ 手动输入内容          │ │
│ │                      │ │
│ └──────────────────────┘ │
│                          │
│ 也可以切回语音录入        │
├──────────────────────────┤
│ 取消           继续分析   │
└──────────────────────────┘
```

说明：

- 手动输入不是“进入完整编辑页”
- 本质仍然属于状态 A

## 4. 状态 B：内容确认态

### 4.1 页面目标

只做一件事：

- 确认这段内容是否正确

### 4.2 低保真结构

```text
┌──────────────────────────┐
│ 关闭        确认内容      │
├──────────────────────────┤
│ 先确认这段话              │
│                          │
│ ┌──────────────────────┐ │
│ │ 识别后的完整文字      │ │
│ │ 可继续修改           │ │
│ └──────────────────────┘ │
│                          │
│ 识别不对？继续改文字      │
│                          │
│ 系统正在准备识别项目      │
├──────────────────────────┤
│ 返回修改       继续分析   │
└──────────────────────────┘
```

### 4.3 模块顺序

从上到下：

1. 顶部栏
2. 一句标题引导
3. 主文本编辑区
4. 一句轻提示
5. 一个状态提示
6. 底部动作区

### 4.4 主次按钮建议

主按钮：

- `继续分析`

次按钮：

- `返回录音`
- 或 `继续修改`

### 4.5 内容区要求

文字区应满足：

- 明显可编辑
- 视觉上是这一屏主角
- 至少能一眼看清 3 到 5 行

不要在这一屏下方塞更多内容。

### 4.6 这一屏不该出现什么

不要出现：

- 候选项目列表
- 完整 AI 摘要卡
- 下一步任务表单
- 时间选择器
- 多个卡片并列

## 5. 状态 C：项目确认态

### 5.1 页面目标

只做一件事：

- 确认这条记录属于哪个项目

### 5.2 低保真结构

```text
┌──────────────────────────┐
│ 返回        确认项目      │
├──────────────────────────┤
│ 这条记录更像属于          │
│                          │
│ ┌──────────────────────┐ │
│ │ AI 首推项目           │ │
│ │ 项目名                │ │
│ │ 客户名 · 当前阶段      │ │
│ └──────────────────────┘ │
│                          │
│ 其他候选                 │
│ [ 候选1 ] [ 候选2 ]       │
│ [ 候选3 ]                │
│                          │
│ 没有合适项目，手动搜索    │
│                          │
│ 一句极短摘要             │
├──────────────────────────┤
│ 稍后再选       确认项目   │
└──────────────────────────┘
```

### 5.3 模块顺序

从上到下：

1. 顶部栏
2. 当前任务标题
3. AI 首推项目
4. 2 到 3 个候选项目
5. 手动搜索入口
6. 一句短摘要
7. 底部动作区

### 5.4 主次按钮建议

主按钮：

- `确认项目`

次按钮：

- `返回修改`
- 或 `稍后再选`

说明：

- 如果允许“暂不关联”，也应该只是一个次动作
- 不能让“更多项目”“确认项目”“继续编辑”都并列抢主动作

### 5.5 项目卡片建议

项目候选必须轻量化：

- 一张首推主卡
- 两三个简版候选
- 更多项目不要首屏全展开

建议字段只保留：

- 项目名
- 客户名
- 阶段
- 一行补充信息

不要在这里展示：

- 太多跟进详情
- 太长的匹配说明
- 多段历史摘要

### 5.6 这一屏不该出现什么

不要完整展开：

- 风险提示
- 阶段建议
- 下一步任务字段
- 时间建议明细
- 长篇 AI 理解

## 6. 状态 D：结果确认态

### 6.1 页面目标

只做一件事：

- 看系统已整理好的结果，决定是否保存

### 6.2 低保真结构

```text
┌──────────────────────────┐
│ 返回        保存前确认    │
├──────────────────────────┤
│ 内容                      │
│ ┌──────────────────────┐ │
│ │ 本次闪录文字          │ │
│ └──────────────────────┘ │
│                          │
│ 关联项目                 │
│ ┌──────────────────────┐ │
│ │ 已确认项目            │ │
│ └──────────────────────┘ │
│                          │
│ 下一步动作               │
│ ┌──────────────────────┐ │
│ │ 建议动作一句话        │ │
│ │ 建议时间              │ │
│ └──────────────────────┘ │
│                          │
│ 查看完整理解             │
├──────────────────────────┤
│ 继续修改      保存本次闪录 │
└──────────────────────────┘
```

### 6.3 模块顺序

从上到下只保留 3 个一级区块：

1. 内容
2. 关联项目
3. 下一步动作

其余能力一律后置折叠。

### 6.4 主次按钮建议

主按钮：

- `保存本次闪录`

次按钮：

- `继续修改`

次级文本入口：

- `查看完整理解`

### 6.5 默认展开与默认收起

默认展开：

- 内容
- 已确认项目
- 下一步动作一句话

默认收起：

- 风险提示
- 阶段建议
- AI 完整摘要
- 详细任务字段
- 额外候选项目

### 6.6 任务区建议

下一步动作区不要直接长成完整表单。

第一眼只给：

- 一句建议动作
- 一个建议时间

需要编辑时，再展开：

- 任务标题
- 日期
- 时间
- 备注

说明：

- 用户要先看到“系统已经帮我补好了什么”
- 不是立刻看到“我还得自己填很多字段”

## 7. 跨状态固定元素建议

### 7.1 顶部标题建议

建议四个状态标题分别为：

- 状态 A：`闪录`
- 状态 B：`确认内容`
- 状态 C：`确认项目`
- 状态 D：`保存前确认`

标题要短，不要长解释。

### 7.2 状态提示建议

顶部或内容区只保留一句轻提示，不要多段说明。

建议风格：

- `先把这次情况记下来`
- `先确认这段话`
- `确认这是哪个项目`
- `系统已经补好下一步`

### 7.3 关闭与放弃入口建议

关闭入口必须始终可达，但不应太抢眼。

建议：

- 顶部左上固定关闭
- 放弃本次闪录不做成大红按钮常驻
- 仅在有内容时，放到次级动作或关闭承接中

## 8. 信息层级建议

### 8.1 一级层

必须始终可见的内容：

- 当前输入内容
- 当前项目结果
- 当前保存主按钮

### 8.2 二级层

按阶段选择性出现：

- 候选项目
- 一句短摘要
- 建议时间

### 8.3 三级层

默认折叠：

- 完整 AI 理解
- 风险提示
- 阶段建议
- 调试信息
- 匹配依据

## 9. 文案长度建议

为了让界面更轻，文案必须明显收短。

建议标准：

- 一级标题：2 到 6 个字
- 轻提示：不超过 16 个字
- 主按钮：4 到 8 个字
- 候选项目说明：一行内解决

不建议：

- 每个模块都有长段解释
- 同一屏出现两句以上帮助文案
- 主按钮文案太长

## 10. 视觉权重建议

即使后面不立刻改视觉，这份低保真也先规定权重。

建议权重顺序：

1. 当前内容输入区
2. 当前主决策区
3. 当前主按钮
4. 次按钮 / 文本入口
5. 说明文字

不要出现：

- 说明文字比输入区更显眼
- AI 模块和输入模块抢同等主视觉
- 每块都像主卡片

## 11. 后续落地建议

如果下一步进入 UI 调整，建议按下面顺序做。

### 第一批

- 先把现有闪录中段切成 4 个状态
- 不先大改样式
- 不先改云函数和数据结构

### 第二批

- 把当前同时展开的模块拆开
- 把项目确认前的 AI 结果压缩

### 第三批

- 把风险、阶段建议、调试信息折叠化
- 把任务详细字段后置展开

### 第四批

- 再统一卡片、按钮、间距和视觉语言

## 12. 验收标准

如果后续照这份低保真推进，至少要满足：

1. 录音后首屏只关注内容，不再堆一页结果
2. 用户会明显经历“内容 -> 项目 -> 保存”的线性过程
3. 项目确认前，不会完整展开大量 AI 内容
4. 保存前页面明显变短，只剩 3 个一级区块
5. 用户始终知道当前这一步只需要做哪一个决定

## 13. 结论

闪录中段后续不应该继续沿着“大页面上不断堆模块”的方向演进。

更合适的方向是：

- 用四个状态分拆中段任务
- 每屏只保留一个主任务
- 把 AI、项目匹配、下一步建议按顺序释放
- 把完整结果后置折叠

这份低保真页面说明可作为后续 UI 调整和前端改造的共同参照。
