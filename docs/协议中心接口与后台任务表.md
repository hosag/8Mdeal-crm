# 协议中心接口与后台任务表

这份文档用于把“隐私政策 / 用户服务协议可在 Web 后台更新”拆成可执行任务。

设计目标：

- Web 后台可编辑、预览、发布协议
- 小程序端可读取当前生效版本
- 用户同意行为可写入 `agreementConsents`

## 一、接口设计原则

- 管理接口均使用 `operatorKey`
- 管理操作写入 `adminAuditLogs`

## 二、建议新增云函数

### 2.1 后台管理接口

#### 1. `adminListLegalDocuments`

用途：

- 后台列表页查询协议
- 查看某类协议的历史版本

请求示例：

```json
{
  "operatorKey": "你的内部管理密钥",
  "docType": "privacy_policy",
  "status": "all",
  "keyword": "",
  "limit": 50
}
```

请求字段建议：

- `operatorKey`
- `docType`
- `status`
- `keyword`
- `limit`

返回示例：

```json
{
  "ok": true,
  "operatorId": "admin_console",
  "total": 2,
  "documents": [
    {
      "docId": "legal_privacy_policy_2026_06_03_v1",
      "docType": "privacy_policy",
      "title": "隐私政策",
      "version": "2026-06-03-v1",
      "status": "published",
      "isCurrent": true,
      "requiresReconsent": true,
      "summary": "补充录音、图片上传、AI 和支付说明",
      "effectiveAt": "2026-06-05T00:00:00.000Z",
      "publishedAt": "2026-06-03T10:00:00.000Z",
      "updatedAt": "2026-06-03T10:00:00.000Z",
      "updatedBy": "admin_console"
    }
  ]
}
```

#### 3. `adminUpsertLegalDocumentDraft`

用途：

- 新建协议草稿
- 更新草稿正文

请求示例：

```json
{
  "operatorKey": "你的内部管理密钥",
  "docId": "",
  "docType": "privacy_policy",
  "title": "隐私政策",
  "version": "2026-06-03-v1",
  "markdownSource": "# 隐私政策\\n\\n这里是正文",
  "summary": "补充录音、AI 和支付说明",
  "changeNotes": [
    "新增语音录入说明",
    "新增 AI 辅助说明"
  ],
  "requiresReconsent": true,
  "effectiveAt": "2026-06-05T00:00:00.000Z",
  "reason": "更新隐私政策草稿"
}
```

入参建议：

- `operatorKey`
- `docId`
- `docType`
- `title`
- `version`
- `markdownSource`
- `summary`
- `changeNotes`
- `requiresReconsent`
- `effectiveAt`
- `reason`

返回示例：

```json
{
  "ok": true,
  "operatorId": "admin_console",
  "action": "created",
  "document": {
    "docId": "legal_privacy_policy_2026_06_03_v1",
    "docType": "privacy_policy",
    "title": "隐私政策",
    "version": "2026-06-03-v1",
    "status": "draft",
    "isCurrent": false,
    "updatedAt": "2026-06-03T09:30:00.000Z",
    "updatedBy": "admin_console"
  }
}
```

实现规则建议：

- 草稿允许多次修改
- 已发布版本默认不允许直接覆盖正文
- 如果 `docId` 为空则创建草稿
- 如果 `docId` 对应的是 `published`，建议直接报错，要求“复制为新版本草稿”

#### 4. `adminPreviewLegalDocument`

用途：

- 把 Markdown 预览成 HTML
- 供后台编辑器实时预览

请求示例：

```json
{
  "operatorKey": "你的内部管理密钥",
  "markdownSource": "# 隐私政策\\n\\n这里是正文"
}
```

返回示例：

```json
{
  "ok": true,
  "operatorId": "admin_console",
  "contentFormat": "markdown",
  "html": "<h1>隐私政策</h1><p>这里是正文</p>"
}
```

说明：

- 第一版如果不想在云端做 Markdown 渲染，也可以在后台前端本地预览
- 但如果要保证渲染结果与发布结果一致，建议最终还是云端统一生成

#### 5. `adminPublishLegalDocument`

用途：

- 把草稿发布成正式版本
- 生成 `htmlSnapshot / plainTextSnapshot / hash`
- 切换同类协议的当前版本

请求示例：

```json
{
  "operatorKey": "你的内部管理密钥",
  "docId": "legal_privacy_policy_2026_06_03_v1",
  "reason": "正式发布隐私政策更新"
}
```

返回示例：

```json
{
  "ok": true,
  "operatorId": "admin_console",
  "action": "published",
  "document": {
    "docId": "legal_privacy_policy_2026_06_03_v1",
    "docType": "privacy_policy",
    "title": "隐私政策",
    "version": "2026-06-03-v1",
    "status": "published",
    "isCurrent": true,
    "requiresReconsent": true,
    "effectiveAt": "2026-06-05T00:00:00.000Z",
    "publishedAt": "2026-06-03T10:00:00.000Z",
    "hash": "sha256:1e0d3d9a..."
  }
}
```

实现规则建议：

1. 校验 `docType / title / version / markdownSource` 完整
2. 生成 `htmlSnapshot`
3. 生成 `plainTextSnapshot`
4. 生成 `hash`
5. 把旧的同 `docType` 的 `isCurrent=true` 版本改为 `false`
6. 当前文档改成：
   - `status = published`
   - `isCurrent = true`
   - `publishedAt = now`
7. 写入 `adminAuditLogs`

#### 6. `adminCloneLegalDocumentDraft`

用途：

- 以当前已发布版本快速创建下一版草稿

请求示例：

```json
{
  "operatorKey": "你的内部管理密钥",
  "sourceDocId": "legal_privacy_policy_2026_06_03_v1",
  "nextVersion": "2026-06-15-v1",
  "reason": "基于当前版本创建下一版草稿"
}
```

返回示例：

```json
{
  "ok": true,
  "operatorId": "admin_console",
  "action": "cloned",
  "document": {
    "docId": "legal_privacy_policy_2026_06_15_v1",
    "docType": "privacy_policy",
    "title": "隐私政策",
    "version": "2026-06-15-v1",
    "status": "draft",
    "isCurrent": false
  }
}
```

说明：

- 这个接口不是第一版绝对必须，但能明显降低后台误操作风险
- 如果不单独做，也可以在 `adminUpsertLegalDocumentDraft` 里支持“复制当前版本生成草稿”

#### 9. `saveAgreementConsent`

用途：

- 用户在小程序里点击“已阅读并同意”后写留痕到 `agreementConsents`

请求示例：

```json
{
  "agreementType": "privacy_policy",
  "version": "2026-06-03-v1",
  "meta": {
    "docType": "privacy_policy",
    "title": "隐私政策",
    "hash": "sha256:1e0d3d9a...",
    "sourcePage": "agreement-center",
    "triggerScene": "first_launch"
  }
}
```

返回示例：

```json
{
  "ok": true,
  "accountId": "acc_xxx",
  "agreementType": "privacy_policy",
  "version": "2026-06-03-v1",
  "acceptedAt": "2026-06-03T10:30:00.000Z"
}
```

实现建议：

- 账号维度留痕
- 同一账号 + 同一协议类型 + 同一版本，允许幂等写入
- 对 `meta.hash` 做保存，确保和发布正文可对齐

#### 10. `getAgreementConsentStatus`

用途：

- 前台判断当前账号是否已同意最新协议
- 为“协议更新后是否要重签”提供状态

请求示例：

```json
{}
```

返回示例：

```json
{
  "ok": true,
  "agreements": [
    {
      "agreementType": "privacy_policy",
      "currentVersion": "2026-06-03-v1",
      "acceptedVersion": "2026-05-20-v1",
      "requiresReconsent": true,
      "isAcceptedCurrent": false
    },
    {
      "agreementType": "user_agreement",
      "currentVersion": "2026-06-03-v1",
      "acceptedVersion": "2026-06-03-v1",
      "requiresReconsent": true,
      "isAcceptedCurrent": true
    }
  ]
}
```

说明：

- 这个接口不是提审前硬门槛，但如果你们后续要做“协议升级后强制重签”，建议第一版就预留

## 七、第一版不建议做的功能

以下内容建议先不要塞进首版：

- 后台富文本拖拽排版
- 协议全文差异高亮 diff
- 多语言协议
- 协议外链公开站点
- 后台多人协作编辑锁
- 自动回滚按钮

先把“能编辑、能发布、能展示、能留痕”做稳，已经足够支撑当前提审和正式商用的基础合规能力。
